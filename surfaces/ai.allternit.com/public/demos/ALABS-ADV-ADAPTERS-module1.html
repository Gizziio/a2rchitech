Provider Adapters & Unified APIs

Learn how Allternit converts chaotic provider streams into a single, predictable ReplyEvent contract — with zero provider-specific code.

📦 @allternit/provider-adapters 🕒 ~25 min 🎯 3 quizzes

1 What Are Provider Adapters?

Every AI provider speaks a slightly different dialect. Anthropic streams content_block_delta events. OpenAI sends choices[0].delta.content. Gemini has its own shape. Your frontend shouldn't care.

Provider adapters are the translation layer. They sit between raw provider streams and the Allternit Replies API, converting every provider's unique event format into a single, canonical ReplyEvent contract.

The Golden Rule: Provider-specific types (like AnthropicStreamChunk or GizziBusEvent) must never leak into the replies-contract package. The contract is pure TypeScript with zero external dependencies.

This package — @allternit/provider-adapters — currently exports the AiSdkReplyAdapter, which handles all AI SDK v6 providers (Anthropic, OpenAI, Gemini, and more) with a single class. No provider-specific branches. No special cases.

The adapter's job is simple but critical: consume a stream of AiSdkStreamPart objects and emit ReplyEvent objects through the onEvent callback. Every downstream consumer — transcripts, reducers, UI components — speaks only ReplyEvent.

const result = streamText({ model, messages, tools });
const adapter = new AiSdkReplyAdapter({
  replyId: "reply_123",
  runId: "run_reply_123",
  onEvent: (event) => {
    // Every event here is a canonical ReplyEvent
    dispatch(event);
  },
});
await adapter.consume(result.fullStream);

We start a text stream from the AI SDK using streamText(). Instead of handling raw provider chunks directly, we create an AiSdkReplyAdapter and give it our reply ID and run ID. Then we tell the adapter to consume the entire stream. For every chunk the provider sends, the adapter figures out what kind of canonical event to emit and calls our onEvent handler. Our app only ever sees clean, typed ReplyEvent objects.

4 Adapter Deep Dive

The AiSdkReplyAdapter is a stateful class. It doesn't just blindly map types — it tracks which items have been opened, buffers text for post-processing, and guards against duplicate events.

Constructor & Options

export interface AiSdkReplyAdapterOptions {
  replyId: string;
  runId: string;
  conversationId?: string;
  onEvent: (event: ReplyEvent) => void;
}

The adapter needs four things to do its job:

  • replyId — the unique ID of this reply (e.g., "reply_123")
  • runId — the run that produced this reply (convention: "run_" + replyId)
  • conversationId — optional, ties the reply to a conversation thread
  • onEvent — the callback where every canonical ReplyEvent is delivered

All events emitted by the adapter include these IDs, so downstream reducers always know which reply and run an event belongs to.

The process() Method

The heart of the adapter is the process(part) method. It takes a single AiSdkStreamPart and routes it through a massive switch statement. Here's how text streaming works:

case "text-start": {
  this.ensureStarted();
  if (!this.openedTextIds.has(part.id)) {
    this.openedTextIds.add(part.id);
    this.emit({
      type: "reply.item.added",
      replyId: this.replyId,
      runId: this.runId,
      itemId: part.id,
      kind: "text",
      ts: this.ts(),
    });
  }
  break;
}

case "text-delta": {
  this.ensureStarted();
  if (!this.openedTextIds.has(part.id)) {
    // Guard: some providers skip text-start, open now
    this.openedTextIds.add(part.id);
    this.emit({ type: "reply.item.added", ... });
  }
  if (part.text) {
    this.textBuffers.set(part.id,
      (this.textBuffers.get(part.id) ?? "") + part.text
    );
    this.emit({
      type: "reply.text.delta",
      itemId: part.id,
      delta: part.text,
      ...
    });
  }
  break;
}

When the provider says text-start, the adapter checks: "Have I already opened this text item?" If not, it emits a reply.item.added event with kind: "text" and records the ID so it won't open it again.

When the provider sends text-delta, the adapter first makes sure the item is open (some providers skip text-start, so this is a safety guard). Then it does two things:

  1. Appends the text to an internal buffer (for post-processing later)
  2. Emits a reply.text.delta event so the UI can stream the text live

The buffer is crucial. When text-end arrives, the adapter uses the accumulated text to extract hidden structures like <thinking> blocks or <document> tags.

6 State Management

The adapter maintains several private data structures to prevent duplicate events and route tool results correctly. Understanding these is key to understanding the adapter's behavior.

openedTextIds
Set<string>
Tracks which text items have already emitted reply.item.added so we never open the same item twice.
openedReasoningIds
Set<string>
Same guard for reasoning items. Prevents duplicate reply.item.added events on reasoning deltas.
pendingTools
Map<string, {...}>
Maps tool-input-start IDs to tool names. Used to correlate partial input with the final tool-call.
toolCallIdToItemId
Map<string, string>
Routes tool-result and tool-error events to the correct item ID, even when providers skip intermediate steps.
textBuffers
Map<string, string>
Accumulates text per item ID. Cleared on text-end and passed to extractStructuredParts.
hasNativeReasoning
boolean
Set to true when native reasoning events arrive. Prevents double-extraction from text buffers.
Why Sets and Maps instead of arrays? Because O(1) lookups matter when you're processing hundreds of stream parts per second. The adapter is designed to be fast enough for real-time streaming UI updates.

7 Post-Processing Magic

Not all providers emit native reasoning or artifact events. Some models (DeepSeek, Kimi, local models) embed thinking inside plain text using XML-like tags. The adapter detects these after a text item closes.

private extractStructuredParts(text: string, afterItemId: string): void {
  // ── Thinking blocks ──
  if (!this.hasNativeReasoning) {
    const thinkRe = /<(?:thinking|think|thought)>([\s\S]*?)(?:<\/(?:thinking|think|thought)>|$)/g;
    // ... emits reply.item.added(reasoning) + reply.reasoning.delta + reply.item.done
  }

  // ── Document blocks ──
  const docRe = /<document(?:\s+title="([^"]*)")?>([\s\S]*?)(?:<\/document>|$)/g;
  // ... emits reply.item.added(artifact) + artifact.created
}

When text-end fires, the adapter looks at the complete accumulated text for that item. It runs two regex passes:

  1. Thinking blocks — matches <thinking>, <think>, or <thought> tags. If the provider already sent native reasoning events, this step is skipped to avoid duplicates.
  2. Document blocks — matches <document title="..."> tags and converts them into artifact items with inline previews.

This is how Allternit supports "extended thinking" on providers that don't natively expose it — without any provider-specific code.

8 consume() & Error Handling

The consume() method wraps the entire stream iteration in a try/catch. This is a deliberate design choice: the adapter never throws. Instead, it emits reply.failed so downstream reducers can update the UI gracefully.

TypeScriptai-sdk.ts
async consume(stream: AsyncIterable<AiSdkStreamPart>): Promise<void> {
  try {
    for await (const part of stream) {
      this.process(part);
    }
  } catch (err) {
    const errMsg = err instanceof Error ? err.message : "stream consume error";
    this.emit({
      type: "reply.failed",
      replyId: this.replyId,
      runId: this.runId,
      error: errMsg,
      ts: this.ts(),
    });
  }
}

This pattern appears throughout the adapter. error parts, abort parts, and finish parts with finishReason: "error" all converge on reply.failed. The frontend only needs to handle one failure event type.

Resilience Principle: Network drops, provider timeouts, and stream corruption are all handled the same way. The adapter absorbs the chaos and emits a clean failure event. This is what makes the Replies API reliable across dozens of provider configurations.

9 Testing Patterns

The test suite uses a simple helper — makeAdapter() — to capture every emitted event into an array. This makes assertions declarative and readable.

TypeScriptai-sdk.test.ts
function makeAdapter(replyId = "r1", runId = "run1") {
  const events: ReplyEvent[] = [];
  const adapter = new AiSdkReplyAdapter({
    replyId, runId,
    onEvent: (e) => events.push(e),
  });
  return { adapter, events };
}

// Example assertion
const { adapter, events } = makeAdapter();
adapter.process({ type: "text-start", id: "t1" });
expect(events[0].type).toBe("reply.item.added");

Key test categories in the suite:

  • Lifecycle — ensures reply.started fires exactly once, and reply.completed / reply.failed fire correctly
  • Text streaming — guards against duplicate reply.item.added, auto-opens items when providers skip text-start
  • Reasoning — verifies native reasoning events produce correct item kinds
  • Tool calls — tests both the tool-input-start path and the fallback tool-call path
  • Sources & files — confirms citations and artifacts are emitted
  • consume() — validates async iteration and graceful error handling
  • Mixed replies — end-to-end sequences with reasoning + text + tools
