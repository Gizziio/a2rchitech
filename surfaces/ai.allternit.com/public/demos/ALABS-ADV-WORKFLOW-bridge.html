Platform Workflow Use Cases

01What Does Allternit Actually Orchestrate?

The Allternit platform doesn't just run workflows in theory — it orchestrates real production workloads across every surface. From the desktop app to the Rails kernel, workflows coordinate multi-step processes with deterministic execution.

🚀 New User Onboarding

When a user signs up, the platform triggers an onboarding workflow that sends a welcome email, provisions a default workspace, initializes profile settings, assigns tutorial content, and invites team members. Each step is a node in a DAG with rollback paths on failure.

💻 Deployment Pipelines

The platform-server build process uses workflows to coordinate Docker image builds, run acceptance tests, update edge configurations, and notify the deployment registry. Condition nodes branch between staging and production paths based on environment flags.

🔄 Data Pipelines

Agent session data flows through ETL workflows: extract conversation logs from SQLite, transform with privacy filters, load into the analytics warehouse. The transform:filter node scrubs PII before the http:request node POSTs to the analytics API.

🤖 Agent Task Orchestration

The Code Mode graph system (seen in graph-code-mode.json) is essentially a workflow DAG. Tasks like CM-K001 through CM-K061 are nodes with blocked_by dependencies. The scheduler executes them in topological order, respecting priority and resource limits.

ℹ️
Real platform file: .allternit/graphs/graph-code-mode.json contains 21 task nodes and 28 edges representing the actual Code Mode implementation graph. Each task_id maps to a WIH (Work Item History) specification that the runner executes as a workflow node.

Custom Node Types

03Platform-Specific Nodes

The base engine provides triggers, transforms, and HTTP nodes. The Allternit platform extends these with custom NodeType registrations that integrate with internal services.

platform/nodes/platform-node-types.ts typescript 
import { NodeType } from '@allternit/workflow-engine';

// Platform-specific node that triggers an agent session
export const triggerAgentNode: NodeType = {
  type: 'platform:trigger-agent',
  category: 'custom',
  displayName: 'Trigger Agent',
  description: 'Start an A2R agent session with a goal',
  inputs: [
    { name: 'goal', type: 'string', required: true },
    { name: 'context', type: 'object' },
  ],
  outputs: [
    { name: 'sessionId', type: 'string' },
    { name: 'status', type: 'string' },
  ],
  configSchema: [
    { name: 'agentModel', type: 'string', default: 'claude-sonnet-4' },
    { name: 'timeout', type: 'number', default: 300000 },
  ],
  executor: async (node, context, inputs) => {
    const model = (node.config?.agentModel as string) || 'claude-sonnet-4';
    const timeout = (node.config?.timeout as number) || 300000;

    const session = await agentRuntime.createSession({
      goal: inputs.goal as string,
      context: inputs.context as Record<string, unknown>,
      model,
      timeout,
      userId: context.userContext?.userId,
    });

    return { sessionId: session.id, status: 'running' };
  },
};

// Platform notification node integrated with the Rails kernel
export const sendNotificationNode: NodeType = {
  type: 'platform:send-notification',
  category: 'custom',
  displayName: 'Send Notification',
  inputs: [
    { name: 'userId', type: 'string', required: true },
    { name: 'message', type: 'string', required: true },
  ],
  outputs: [{ name: 'delivered', type: 'boolean' }],
  executor: async (_node, _context, inputs) => {
    const res = await fetch('https://kernel.allternit.local/v1/notify', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        user_id: inputs.userId,
        body: inputs.message,
      }),
    });
    return { delivered: res.ok };
  },
};

// Database update node that writes to the platform Postgres
export const updateDatabaseNode: NodeType = {
  type: 'platform:update-database',
  category: 'database',
  displayName: 'Update Database',
  configSchema: [
    { name: 'table', type: 'string', required: true },
    { name: 'operation', type: 'string', default: 'upsert' },
  ],
  executor: async (node, _context, inputs) => {
    const table = node.config?.table as string;
    const operation = (node.config?.operation as string) || 'upsert';

    const result = await db.query(
      `SELECT ${operation}_record($1, $2)`,
      [table, JSON.stringify(inputs.data)],
    );

    return { rowsAffected: result.rowCount };
  },
};

Scheduler Integration

05How the Platform Handles Workloads

The Allternit platform runs two scheduler instances: one for user-facing workflows (onboarding, permission requests) and one for background workloads (analytics ETL, agent session cleanup). They share the same createScheduler factory but with different configurations.

platform/scheduler/platform-schedulers.ts typescript 
import { createScheduler } from '@allternit/workflow-engine';

// User-facing scheduler: low latency, high priority, small queue
export const userScheduler = createScheduler({
  maxConcurrency: 10,
  queueLimit: 50,
  defaultTimeout: 30000,       // 30s — users won't wait longer
  enablePriority: true,
});

// Background scheduler: high throughput, lower priority, large queue
export const backgroundScheduler = createScheduler({
  maxConcurrency: 25,
  queueLimit: 500,
  defaultTimeout: 300000,      // 5min — ETL can take time
  enablePriority: false,       // FIFO for background jobs
});

// Submit onboarding workflow to user queue with high priority
await userScheduler.submit({
  id: 'task-onboarding-usr_9a3f',
  priority: 10,               // Bump ahead of analytics
  node: { id: 'onboarding-root', type: 'trigger:manual', name: 'Onboarding' },
  execution: onboardingExecution,
  timeout: 30000,
});

// Submit nightly analytics rollup to background queue
await backgroundScheduler.submit({
  id: 'task-analytics-nightly',
  node: { id: 'analytics-root', type: 'trigger:schedule', name: 'Nightly Rollup' },
  execution: analyticsExecution,
  timeout: 300000,
});

⚡ Why Two Queues?

User-facing workflows must complete before HTTP response timeouts. If onboarding took 5 minutes because it was stuck behind 200 analytics jobs, users would abandon the signup flow. Priority sorting ensures critical paths run first.

Background workflows can tolerate backpressure. The analytics ETL running at 02:00 has no human waiting. A larger queue absorbs spikes without dropping jobs, and FIFO ordering prevents starvation of long-running tasks.

Observability

06Monitoring Workflow Health

The platform monitors workflow health through engine hooks, scheduler status polling, and structured execution traces. Every execution emits metrics that feed into the internal observability stack.

platform/observability/workflow-metrics.ts typescript 
import { WorkflowHooks } from '@allternit/workflow-engine';

export const observabilityHooks: WorkflowHooks = {
  beforeExecute: async (execution) => {
    metrics.increment('workflow.started', {
      workflow_id: execution.workflowId,
      tier: 'adv',
    });
    logger.info('Workflow started', { executionId: execution.id });
  },

  afterExecute: async (execution) => {
    const duration = new Date(execution.completedAt!).getTime() -
                     new Date(execution.startedAt!).getTime();

    metrics.histogram('workflow.duration_ms', duration, {
      workflow_id: execution.workflowId,
      status: execution.status,
    });

    metrics.increment('workflow.completed', {
      status: execution.status,
    });
  },

  beforeNodeExecute: async (node, _context) => {
    metrics.increment('workflow.node.started', {
      node_type: node.type,
    });
  },

  afterNodeExecute: async (node, _context, result) => {
    metrics.increment('workflow.node.completed', {
      node_type: node.type,
    });
    span.setAttribute('node.result', JSON.stringify(result).slice(0, 1000));
  },

  onError: async (error, context) => {
    metrics.increment('workflow.node.failed', {
      error_code: error.code,
      node_id: error.nodeId || 'unknown',
    });

    logger.error('Workflow node failed', {
      code: error.code,
      message: error.message,
      nodeId: error.nodeId,
      executionPath: context.state.executionPath,
    });

    // Alert on critical workflow failures
    if (error.code === 'NODE_EXECUTION_ERROR') {
      await alerting.send({
        severity: 'warning',
        title: `Workflow node failed: ${error.nodeId}`,
        context: context.state.executionPath,
      });
    }
  },
};

// Attach hooks to the engine
const engine = createWorkflowEngine({
  hooks: observabilityHooks,
  maxConcurrentExecutions: 50,
});

Capstone Project

08Build an Onboarding Workflow

Design a 6-node workflow for new user onboarding that includes: email welcome, profile setup, tutorial assignment, team invite, and analytics tracking. Add error handling for the email node when SMTP is down.

🏆 Requirements

  • Node 1: trigger:manual — accepts userId, email, teamId (optional)
  • Node 2: transform:map — builds profile object with defaults
  • Node 3: http:request — sends welcome email via platform API (timeout: 15s, retries: 3)
  • Node 4: platform:assign-tutorial — assigns "getting-started" tutorial
  • Node 5: condition:if — if teamId present, branch to invite node
  • Node 6: platform:analytics — tracks onboarding completion event
  • Error handling: If email fails after retries, log the failure and continue to analytics (degraded completion)
capstone/onboarding-workflow.ts typescript 
import { createWorkflowEngine } from '@allternit/workflow-engine';
import { sendNotificationNode, updateDatabaseNode } from './platform-node-types';

const engine = createWorkflowEngine({
  maxConcurrentExecutions: 10,
  defaultTimeout: 60000,
  hooks: {
    onError: async (error) => {
      console.error('[Onboarding] Node failed:', error.message);
    },
  },
});

// Register platform custom nodes
engine.registerNodeType(sendNotificationNode);
engine.registerNodeType(updateDatabaseNode);

// Define the onboarding workflow
const onboardingWorkflow = {
  id: 'capstone-onboarding',
  name: 'Capstone: New User Onboarding',
  version: '1.0.0',
  nodes: [
    {
      id: 'trigger',
      type: 'trigger:manual',
      name: 'New User Signup',
    },
    {
      id: 'build-profile',
      type: 'transform:map',
      name: 'Build Profile',
      config: {
        expression: `{
          userId: data.userId,
          email: data.email,
          displayName: data.email.split('@')[0],
          createdAt: new Date().toISOString()
        }`,
      },
      inputs: [{ target: 'data', source: '${inputs}' }],
    },
    {
      id: 'send-welcome',
      type: 'platform:send-notification',
      name: 'Send Welcome Email',
      inputs: [
        { target: 'userId', source: '${inputs.userId}' },
        { target: 'message', source: 'Welcome to Allternit!' },
      ],
    },
    {
      id: 'assign-tutorial',
      type: 'platform:assign-tutorial',
      name: 'Assign Tutorial',
      config: { tutorialId: 'getting-started' },
      inputs: [{ target: 'userId', source: '${inputs.userId}' }],
    },
    {
      id: 'check-team',
      type: 'condition:if',
      name: 'Has Team ID?',
      config: { condition: 'data.teamId != null' },
      inputs: [{ target: 'data', source: '${inputs}' }],
    },
    {
      id: 'invite-team',
      type: 'platform:update-database',
      name: 'Create Team Invite',
      config: { table: 'team_invites', operation: 'insert' },
      inputs: [{ target: 'data', source: '${inputs}' }],
    },
    {
      id: 'track-analytics',
      type: 'platform:analytics',
      name: 'Track Completion',
      config: { event: 'capstone.onboarding.completed' },
      inputs: [{ target: 'userId', source: '${inputs.userId}' }],
    },
    {
      id: 'output',
      type: 'output:result',
      name: 'Onboarding Result',
    },
  ],
  connections: [
    { id: 'c1', source: 'trigger', target: 'build-profile' },
    { id: 'c2', source: 'build-profile', target: 'send-welcome' },
    { id: 'c3', source: 'send-welcome', target: 'assign-tutorial' },
    { id: 'c4', source: 'assign-tutorial', target: 'check-team' },
    { id: 'c5', source: 'check-team', target: 'invite-team', condition: 'result.true' },
    { id: 'c6', source: 'check-team', target: 'track-analytics', condition: 'result.false' },
    { id: 'c7', source: 'invite-team', target: 'track-analytics' },
    { id: 'c8', source: 'track-analytics', target: 'output' },
  ],
  errorHandling: {
    retryCount: 3,
    retryDelay: 2000,
    continueOnError: true,   // Degraded completion: analytics still tracked
    maxErrors: 2,
  },
};

engine.registerWorkflow(onboardingWorkflow);

// Execute with test input
const result = await engine.execute('capstone-onboarding', {
  userId: 'usr_test_001',
  email: 'dev@allternit.local',
  teamId: 'team_eng',
});

console.log('Execution status:', result.status);
console.log('Execution path:', result.context?.state.executionPath);
capstone/onboarding-workflow.test.ts typescript 
import { describe, it, expect, vi } from 'vitest';
import { createWorkflowEngine } from '@allternit/workflow-engine';

describe('Capstone: Onboarding Workflow', () => {
  it('should complete full onboarding with team invite', async () => {
    const engine = createWorkflowEngine();
    // ... register nodes and workflow ...

    const execution = await engine.execute('capstone-onboarding', {
      userId: 'usr_001',
      email: 'test@example.com',
      teamId: 'team_001',
    });

    expect(execution.status).toBe('completed');
    expect(execution.context?.state.executionPath).toEqual([
      'trigger', 'build-profile', 'send-welcome',
      'assign-tutorial', 'check-team', 'invite-team',
      'track-analytics', 'output',
    ]);
  });

  it('should skip team invite when no teamId provided', async () => {
    const execution = await engine.execute('capstone-onboarding', {
      userId: 'usr_002',
      email: 'solo@example.com',
    });

    expect(execution.context?.state.executionPath).not.toContain('invite-team');
    expect(execution.context?.state.executionPath).toContain('track-analytics');
  });

  it('should handle SMTP failure with degraded completion', async () => {
    // Mock notification node to throw SMTP error
    engine.registerNodeType({
      type: 'platform:send-notification',
      category: 'custom',
      displayName: 'Failing SMTP',
      executor: async () => {
        throw new Error('SMTP connection refused');
      },
    });

    const execution = await engine.execute('capstone-onboarding', {
      userId: 'usr_003',
      email: 'bounce@example.com',
    });

    // With continueOnError: true, workflow completes despite email failure
    expect(execution.status).toBe('completed');
    expect(execution.context?.state.failedNodes).toContain('send-welcome');
    expect(execution.context?.state.executionPath).toContain('track-analytics');
  });
});
ℹ️
Key design decision: Setting continueOnError: true in errorHandling means the onboarding workflow completes even if the welcome email fails. The analytics event is still tracked, the profile is still created, and the user can proceed. The failure is captured in failedNodes for later investigation.
