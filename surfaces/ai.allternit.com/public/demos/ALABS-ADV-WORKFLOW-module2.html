01 — The Problem

01Why Promise.all() Doesn't Scale

When you kick off a workflow with dozens of nodes, the naive approach is to fire every task at once and hope for the best. This breaks down in production for three reasons:

1. Resource Exhaustion

Opening 200 HTTP connections simultaneously will hit OS file-descriptor limits, crash the event loop, or get rate-limited by downstream APIs.

2. No Backpressure

Once Promise.all() is invoked, there is no mechanism to slow down or pause incoming work. Memory grows unbounded as tasks queue up in the microtask queue.

3. Cancellation is Impossible

There is no built-in way to abort an individual promise mid-flight. You need AbortController plumbing, which Promise.all() does not manage for you.

⚠
Production reality: A workflow engine without a scheduler is just a script. The scheduler is what makes workflows safe under load.

The Allternit Workflow Engine solves this with a dedicated Scheduler subsystem. It sits between the DAG traversal logic and the actual node execution, enforcing limits, managing lifecycles, and giving you observability hooks.

03 — Concurrency

03Concurrency Control & Backpressure

The heart of the scheduler is processQueue(). It is called every time a task is submitted and again in the finally block when a task finishes, creating a self-sustaining pipeline.

scheduler/index.ts — processQueue()
async function processQueue(): Promise<void> {
  // Gate 1: paused
  // Gate 2: at max concurrency
  // Gate 3: nothing to do
  if (isPaused || running.size >= maxConcurrency || queue.length === 0) {
    return;
  }

  // Sort by priority if enabled (descending)
  if (enablePriority) {
    queue.sort((a, b) => (b.priority || 0) - (a.priority || 0));
  }

  const task = queue.shift();
  if (!task) return;

  const controller = new AbortController();
  running.set(task.id, controller);

  try {
    const timeout = task.timeout || defaultTimeout;
    const timeoutId = setTimeout(() => controller.abort(), timeout);

    const result = await executeTask(task, controller.signal);

    clearTimeout(timeoutId);
    completed++;
  } catch (error) {
    failed++;
    throw error;
  } finally {
    running.delete(task.id);
    // Recursively process the next slot
    processQueue();
  }
}

Gate Logic Explained

The three early-return conditions form a precise state machine:

  • isPaused — The scheduler is in a paused state. New tasks queue up but are not dequeued until resume() is called.
  • running.size >= maxConcurrency — The concurrency budget is exhausted. This is the backpressure mechanism.
  • queue.length === 0 — Nothing is waiting. The pipeline stalls until submit() is called again.

The queueLimit Safety Valve

Before a task ever enters the queue, submit() checks queue.length >= queueLimit and throws. This prevents unbounded memory growth when the system is under sustained load and consumers are slower than producers.

scheduler/index.ts — submit()
async function submit(task: Task): Promise<unknown> {
  if (queue.length >= queueLimit) {
    throw new Error('Queue limit reached');
  }
  queue.push(task);
  return processQueue();
}
04 — Cancellation

04Cancellation & Timeouts

The scheduler uses the standard Web AbortController API for cooperative cancellation. Every task in the running Map owns its own controller. Timeouts are implemented as automatic aborts.

scheduler/index.ts — Timeout wiring
const timeout = task.timeout || defaultTimeout;
const timeoutId = setTimeout(() => controller.abort(), timeout);

const result = await executeTask(task, controller.signal);

clearTimeout(timeoutId);

If the task finishes before the timeout, clearTimeout prevents the abort from firing. If the timeout fires first, controller.signal transitions to aborted, and the executor can react:

scheduler/index.ts — Task executor
async function executeTask(
  task: Task,
  signal: AbortSignal
): Promise<unknown> {
  if (signal.aborted) {
    throw new Error('Task cancelled');
  }

  // Real integration would pass signal to fetch, fs, etc.
  return { taskId: task.id, status: 'completed' };
}
⚠
Cooperative cancellation: The signal only works if the underlying executor checks signal.aborted or passes the signal to an abort-aware API like fetch(). Non-cooperative code (e.g. a tight while loop) will ignore the abort.

Cancelling a Queued vs Running Task

The cancel() method handles both cases with a simple branch:

scheduler/index.ts — cancel()
function cancel(taskId: string): boolean {
  // Case 1: Task is currently running
  const controller = running.get(taskId);
  if (controller) {
    controller.abort();
    return true;
  }

  // Case 2: Task is still queued
  const index = queue.findIndex(t => t.id === taskId);
  if (index >= 0) {
    queue.splice(index, 1);
    return true;
  }

  return false; // Not found anywhere
}

Key Difference

A running task is aborted via AbortController.abort(). The task may still briefly execute cleanup in its finally block. A queued task is simply removed from the array — it never starts, so no abort signal is needed.

05 — Priority

05Priority Queues & Pause/Resume

When enablePriority: true is set in the config, the scheduler sorts the queue every time processQueue() runs. The sort is descending by priority — higher numbers execute first.

Priority sort in processQueue()
if (enablePriority) {
  queue.sort((a, b) => (b.priority || 0) - (a.priority || 0));
}

This is a stable-enough approach for workflow use cases because:

  • The queue size is bounded by queueLimit (default 100).
  • Sorting runs only when a slot frees up, not on every insertion.
  • Tasks with the same priority retain their relative order (stable sort in V8).

Pause / Resume State Machine

The scheduler supports a global pause. This is distinct from cancelling — paused tasks remain in the running Map and continue their work. Only new tasks from the queue are blocked.

Pause & Resume
function pause(): void {
  isPaused = true;
}

function resume(): void {
  isPaused = false;
  processQueue(); // Kick the pipeline
}

06 — Integration

06Engine Integration: DAG to Scheduler

The workflow engine traverses the DAG depth-first. After a node completes successfully, it walks the outgoing connections and recursively executes downstream nodes. But where does the scheduler fit in?

ⓘ
In the current implementation, executeNode awaits the executor directly. The scheduler is designed to wrap this call, turning synchronous-looking node execution into managed, cancellable, concurrency-limited tasks.
engine/workflow-engine.ts — executeNode()
async function executeNode(
  node: WorkflowNode,
  execution: WorkflowExecution,
  dag: { nodes: WorkflowNode[]; connections: Connection[] }
): Promise<void> {

  // 1. Skip if already executed
  if (execution.context!.state.completedNodes.includes(node.id)) {
    return;
  }

  // 2. Verify all dependencies have results
  const incoming = dag.connections.filter(c => c.target === node.id);
  const depsMet = incoming.every(c =>
    execution.context!.state.completedNodes.includes(c.source) ||
    execution.context!.state.nodeResults[c.source] !== undefined
  );
  if (!depsMet) return;

  // 3. Mark active
  execution.context!.state.activeNodes.push(node.id);
  execution.context!.state.executionPath.push(node.id);

  // 4. Resolve inputs from upstream results
  const inputs: Record<string, unknown> = {};
  incoming.forEach(conn => {
    const sourceResult = execution.context!.state.nodeResults[conn.source];
    if (sourceResult) {
      inputs[conn.sourcePort || 'data'] = sourceResult;
    }
  });

  // 5. Execute via node type executor
  const nodeType = nodeTypes.get(node.type)!;
  const result = await nodeType.executor(node, execution.context!, inputs);

  // 6. Store result
  execution.context!.state.nodeResults[node.id] = result;
  execution.context!.state.completedNodes.push(node.id);

  // 7. Traverse outgoing connections
  const outgoing = dag.connections.filter(c => c.source === node.id);
  for (const conn of outgoing) {
    if (conn.condition) {
      const met = evaluateCondition(conn.condition, result, execution.context!);
      if (!met) continue;
    }
    const nextNode = dag.nodes.find(n => n.id === conn.target);
    if (nextNode) {
      await executeNode(nextNode, execution, dag);
    }
  }
}

The integration point is step 5. Instead of awaiting the executor directly, the engine would construct a Task and call scheduler.submit(task). The scheduler then manages when that task actually begins based on concurrency limits, priority, and pause state.

Future Integration Pattern

const task: Task = {

07 — Errors

07Error Handling & Recovery

Errors can happen at three layers: the scheduler, the engine's node executor, and the workflow's DAG traversal. Each layer has its own recovery semantics.

Scheduler Layer

When executeTask throws, the scheduler catches it, increments failed, and re-throws. The re-throw is important — it allows the caller (the engine) to decide whether to fail the entire workflow or continue.

scheduler/index.ts — Error propagation
try {
  const result = await executeTask(task, controller.signal);
  clearTimeout(timeoutId);
  completed++;
} catch (error) {
  failed++;
  throw error; // Propagate to the engine
} finally {
  running.delete(task.id);
  processQueue();
}

Engine Layer

The engine catches node errors in executeNode, records them in the execution state, and propagates them up to execute() where the workflow-level try/catch marks the entire execution as failed and triggers the onError hook.

engine/workflow-engine.ts — Node error handling
try {
  const result = await nodeType.executor(node, execution.context!, inputs);
  // ... store result ...
} catch (error) {
  nodeExecution.status = 'failed';
  nodeExecution.error = {
    code: 'NODE_EXECUTION_ERROR',
    message: error instanceof Error ? error.message : String(error),
    nodeId: node.id,
  };
  execution.context!.state.failedNodes.push(node.id);
  throw error; // Fail the workflow
}

Hooks for Observability

The engine config accepts a hooks object with lifecycle callbacks. These are perfect for logging, metrics, and external alerting:

types.ts — WorkflowHooks
export interface WorkflowHooks {
  beforeExecute?: (execution: WorkflowExecution) => Promise<void>;
  afterExecute?: (execution: WorkflowExecution) => Promise<void>;
  beforeNodeExecute?: (node: WorkflowNode, context: ExecutionContext) => Promise<void>;
  afterNodeExecute?: (node: WorkflowNode, context: ExecutionContext, result: unknown) => Promise<void>;
  onError?: (error: ExecutionError, context: ExecutionContext) => Promise<void>;
}

Where Retry Logic Belongs

Retry is currently not implemented in the scheduler. The ideal location is inside executeNode (engine layer) or as a wrapper around the scheduler's submit(). The NodeExecution type already has a retryCount field, signaling that retries are part of the intended design.

08 — Knowledge Check

08Interactive Quizzes

1. What happens when running.size >= maxConcurrency inside processQueue()?
Correct! The scheduler uses early return as a backpressure gate. Tasks remain in the queue until a running task finishes and frees a concurrency slot.
Not quite. The scheduler never preempts running tasks. It simply waits for a slot to open via the finally block's recursive processQueue() call.
2. How does task cancellation differ for a queued task vs a running task?
Correct! Running tasks need cooperative cancellation via AbortController. Queued tasks have not started, so removing them from the queue is sufficient.
Not quite. Look at the cancel() implementation: it branches on whether the task ID exists in the running Map. Queued tasks don't have a controller yet.
3. In the engine's executeNode(), when are outgoing connections traversed?
Correct! Outgoing connections are traversed inside the try block, after the result is stored. If the node fails, the catch throws before downstream nodes are reached.
Not quite. Outgoing traversal happens after storing the result and calling afterNodeExecute hooks. It's in the success path only.
09 — Capstone

09Capstone: Build a Retry-Aware Scheduler Wrapper

Objective

Wrap createScheduler() with exponential backoff retry logic. Track retry counts per task, emit an onRetry hook, and test with a task that fails twice before succeeding.

The Solution

retry-scheduler.ts
import { createScheduler, type Scheduler, type Task } from './scheduler';

export interface RetryConfig {
  maxRetries?: number;
  baseDelay?: number;
  maxDelay?: number;
  onRetry?: (taskId: string, attempt: number, delay: number) => void;
}

export interface RetryAwareScheduler extends Scheduler {
  getRetryCount(taskId: string): number;
}

export function createRetryScheduler(
  retryConfig: RetryConfig = {}
): RetryAwareScheduler {
  const {
    maxRetries = 3,
    baseDelay = 500,
    maxDelay = 10000,
    onRetry,
  } = retryConfig;

  const scheduler = createScheduler();
  const retryCounts = new Map<string, number>();

  function exponentialBackoff(attempt: number): number {
    const delay = baseDelay * (2 ** attempt);
    return Math.min(delay, maxDelay);
  }

  async function submitWithRetry(task: Task): Promise<unknown> {
    try {
      return await scheduler.submit(task);
    } catch (error) {
      const count = retryCounts.get(task.id) || 0;

      if (count >= maxRetries) {
        retryCounts.delete(task.id);
        throw error;
      }

      const nextCount = count + 1;
      retryCounts.set(task.id, nextCount);

      const delay = exponentialBackoff(nextCount);
      onRetry?.(task.id, nextCount, delay);

      await new Promise(resolve => setTimeout(resolve, delay));

      // Clone task for retry to avoid mutating original state
      return submitWithRetry({ ...task });
    }
  }

  return {
    ...scheduler,
    submit: submitWithRetry,
    submitAll: (tasks: Task[]) =>
      Promise.all(tasks.map(t => submitWithRetry(t))),
    getRetryCount(taskId: string) {
      return retryCounts.get(taskId) || 0;
    },
  };
}

Walkthrough

  • Closure state: retryCounts is a Map<string, number> scoped inside the factory. It persists across calls but is hidden from consumers.
  • Exponential backoff: baseDelay × 2attempt capped at maxDelay. Attempt 1 = 500ms, attempt 2 = 1000ms, attempt 3 = 2000ms.
  • Hook: onRetry fires before the delay, giving the caller a chance to log or emit metrics.
  • Task cloning: We spread the task object on retry so any downstream mutations don't leak back to the caller's reference.
