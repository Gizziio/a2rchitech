The Challenge

76 Templates. Zero Reach.

Allternit has 76 powerful templates. But Claude can't see them. Cursor ignores them. Codex doesn't know they exist. Every platform speaks a different dialect.

Platform Fragmentation

Each LLM tool uses its own protocol: MCP for Claude, CLI for Codex, extensions for VS Code. Building separate integrations is unsustainable.

Duplicated Effort

Without a universal layer, every template needs 5+ custom implementations. One bug fix becomes five. One feature lags on half the platforms.

The SDK Answer

One PluginHost interface. One manifest. One execution path. The adapters handle the translation to every platform automatically.

Core Innovation

The PluginHost Contract

Every plugin interacts with the world through a single, predictable interface. No matter if it's running inside Claude Desktop or a Python LangChain agent, the code stays identical.

types.ts
export interface PluginHost { // LLM access: complete, stream, count tokens readonly llm: LlmCapabilities; // Tool registry: check availability and execute readonly tools: ToolRegistry; // UI rendering: markdown, code, images, charts readonly ui: UiCapabilities; // Key/value storage persistence readonly storage: StorageBackend; // Session context and conversation history readonly session: SessionContext; // Structured logging readonly logger: Logger; }

Think of PluginHost as a power strip with six sockets. Your plugin plugs into it and gets electricity (LLM), tools (drill, saw), a display (monitor), memory (hard drive), session info (who's using it), and a logger (diagnostics).

No matter which house (platform) the power strip is in, your appliance works exactly the same way. The adapter is just the wall plug shape converter.

Hands-On

Building Your First Universal Plugin

Let's build a minimal market-research plugin. The same code will work on Claude, VS Code, Codex, and any HTTP client.

src/index.ts
import { PluginHost } from '@allternit/plugin-sdk'; export async function run(host: PluginHost, params: any) { // 1. Ask the LLM to analyze the topic const result = await host.llm.complete( `Analyze market trends for: ${params.topic}` ); // 2. Render the result as markdown host.ui.renderMarkdown(result); // 3. If we have a browser, use it for live data if (host.tools.has('browser_navigate')) { return host.tools.execute('browser_navigate', { url: params.url }); } // 4. Otherwise fall back to LLM knowledge return host.llm.complete(`Research: ${params.url}`); }

Step 1: We receive a host object and user params. The host is our universal power strip.

Step 2: We call host.llm.complete() to ask the AI to analyze market trends. We don't care if the underlying model is Claude, GPT-4, or Copilot.

Step 3: We use host.ui.renderMarkdown() to display the result. On VS Code this opens a webview panel; in CLI it prints to terminal; in Claude it returns text.

Step 4: We check if browser_navigate is available. If yes, we browse live. If not, we gracefully fall back to the LLM's training data.

manifest.json
{ "id": "market-research", "name": "Market Research", "version": "1.0.0", "runtime": "allternit-plugin-v1", "category": "research", "requires": { "llm": { "capabilities": ["reasoning"] }, "tools": ["web_search", { "tool": "browser_navigate", "required": false }] }, "provides": { "functions": [{ "name": "research", "description": "Research a market topic", "parameters": { "type": "object", "properties": { "topic": { "type": "string", "description": "Topic to research" } }}, "returns": { "type": "object" } }] } }
Resilience

Graceful Degradation

Not every platform has every tool. The SDK is designed so plugins adapt instead of crashing. The same logic runs on a super-powered HTTP server or a minimal MCP client.

Capstone Project

Build a Cross-Platform Code Review Plugin

Put everything together. Your mission is to create a plugin that reads a pull request diff, analyzes it with an LLM, and posts suggestions — no matter which platform is hosting it.

Project Requirements

  • Define a PluginManifest with requires for file_read and llm.reasoning.
  • Implement the plugin using PluginHost (llm.complete, tools.has, ui.renderMarkdown).
  • Add graceful degradation: if file_read is unavailable, accept the diff as a string parameter.
  • Test locally with the CLI adapter by passing a sample diff via command-line arguments.
  • Run it with the HTTP adapter and call it via curl.
  • Bonus: package it as an .allp file and document the MCP configuration for Claude Desktop.
