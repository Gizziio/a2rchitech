Resilience Patterns.
Rate Limiting, Failover & Circuit Breakers.

Building production-grade AI provider adapters means embracing failure as the default. Learn to shield your AiSdkReplyAdapter from cascading outages with token buckets, exponential backoff, and circuit breakers.

01 The Resilience Problem

External APIs fail. It is not a matter of if but when. The AiSdkReplyAdapter consumes an AsyncIterable<AiSdkStreamPart> from providers like OpenAI, Anthropic, and Gemini. Any point in that pipeline can fail:

Failure Modes in Production
FailureHTTP StatusImpact
Rate limiting429 Too Many RequestsTemporarily blocked; retry after header
Server errors500/502/503Provider is degraded or down
Network partitionsTimeout / ECONNRESETRequest never reaches the provider
Stream abortsPartial bodyClient receives incomplete deltas
Token exhaustion400 or stream errorModel refuses to continue

Without resilience patterns, a single provider hiccup becomes a user-facing outage. The adapter layer is the runtime boundary between the Allternit kernel and external vendors — it is exactly where these defenses belong.

Architectural principle: Layer 3-adapters/contracts/runtime.ts defines ToolExecutionRequest with retryCount and idempotencyKey. Resilience is not an afterthought; it is a first-class contract concern.

02 Rate Limiting

Before you even send a request, you should know whether you are allowed to. Rate limiting prevents outbound storms and keeps your adapter a well-behaved citizen of the provider's infrastructure.

Token Bucket vs. Sliding Window

AlgorithmMechanismProsCons
Token BucketTokens refill at a fixed rate; each request consumes oneBurst-friendly, simple stateCan burst unexpectedly at window boundaries
Sliding WindowCounts requests in a rolling time windowSmoother distributionHigher memory cost (per-request timestamps)
Fixed WindowResets counter at interval boundariesMinimal stateThundering herd at reset edge

For AI provider adapters, Token Bucket is usually the right trade-off: it allows short bursts (useful for parallel tool calls) while capping sustained throughput.

Implementing a TokenBucket in TypeScript

TypeScript TokenBucket.ts
interface TokenBucketOptions {
  capacity: number;      // max tokens in the bucket
  refillRate: number;    // tokens added per second
  refillIntervalMs?: number;
}

export class TokenBucket {
  private tokens: number;
  private lastRefill: number;
  private capacity: number;
  private refillRate: number;
  private intervalMs: number;

  constructor(opts: TokenBucketOptions) {
    this.capacity = opts.capacity;
    this.refillRate = opts.refillRate;
    this.intervalMs = opts.refillIntervalMs ?? 1000;
    this.tokens = opts.capacity;
    this.lastRefill = Date.now();
  }

  private refill(): void {
    const now = Date.now();
    const elapsed = now - this.lastRefill;
    const tokensToAdd = Math.floor(
      (elapsed / this.intervalMs) * this.refillRate
    );
    if (tokensToAdd > 0) {
      this.tokens = Math.min(this.capacity, this.tokens + tokensToAdd);
      this.lastRefill = now;
    }
  }

  /** Returns true if a token was consumed. */
  consume(): boolean {
    this.refill();
    if (this.tokens >= 1) {
      this.tokens -= 1;
      return true;
    }
    return false;
  }

  /** Estimated wait time in ms until next token is available. */
  waitTimeMs(): number {
    this.refill();
    if (this.tokens >= 1) return 0;
    const needed = 1 - this.tokens;
    return Math.ceil((needed / this.refillRate) * this.intervalMs);
  }
}

Integrating with the Adapter Layer

TypeScript RateLimitedAdapter.ts
import { AiSdkReplyAdapter, AiSdkStreamPart } from "@allternit/provider-adapters";
import { TokenBucket } from "./TokenBucket";

export class RateLimitedAiSdkAdapter extends AiSdkReplyAdapter {
  private bucket = new TokenBucket({ capacity: 60, refillRate: 10 });

  async consume(stream: AsyncIterable<AiSdkStreamPart>): Promise<void> {
    if (!this.bucket.consume()) {
      const wait = this.bucket.waitTimeMs();
      await new Promise(r => setTimeout(r, wait));
    }
    return super.consume(stream);
  }
}
Design note: The @allternit/replies-contract ToolExecutionRequest already carries retryCount and idempotencyKey. When rate limiting triggers, increment retryCount and preserve the idempotency key so upstream retries remain safe.

03 Exponential Backoff

Linear retry (wait 1s, wait 1s, wait 1s...) is dangerous. During an incident, every client retries at the same cadence, creating a thundering herd that prolongs recovery. Exponential backoff spreads retries across time.

Why Linear Retry Fails

Linear: 1s + 1s + 1s
All clients collide
Exponential: 1s + 2s + 4s + 8s
Retries desynchronize

Jitter Strategies

Even exponential backoff can synchronize if all clients start at the same time. Jitter adds randomness to break alignment:

  • Full jitter: sleep = random(0, base * 2^attempt) — maximum desynchronization, but can be very fast
  • Equal jitter: sleep = (base * 2^attempt) / 2 + random(0, base * 2^attempt / 2) — balances spread and minimum delay
  • Decorrelated jitter: sleep = min(cap, random(base, sleep_prev * 3)) — AWS approach; excellent in practice

Implementation

TypeScript backoff.ts
interface BackoffOptions {
  baseMs: number;
  maxDelayMs: number;
  maxRetries: number;
  jitter: "none" | "full" | "equal" | "decorrelated";
}

export class ExponentialBackoff {
  private prevSleep = 0;

  constructor(private opts: BackoffOptions) {}

  delayMs(attempt: number): number {
    const base = this.opts.baseMs * (Math.pow(2, attempt));
    let sleep: number;

    switch (this.opts.jitter) {
      case "full":
        sleep = Math.random() * Math.min(base, this.opts.maxDelayMs);
        break;
      case "equal":
        sleep = Math.min(base, this.opts.maxDelayMs) / 2
          + Math.random() * (Math.min(base, this.opts.maxDelayMs) / 2);
        break;
      case "decorrelated":
        sleep = Math.min(
          this.opts.maxDelayMs,
          Math.random() * this.opts.baseMs + this.prevSleep * 3
        );
        break;
      default:
        sleep = Math.min(base, this.opts.maxDelayMs);
    }

    this.prevSleep = sleep;
    return Math.floor(sleep);
  }

  async execute<T>(
    fn: () => Promise<T>,
    isRetryable: (err: unknown) => boolean
  ): Promise<T> {
    for (let attempt = 0; attempt <= this.opts.maxRetries; attempt++) {
      try {
        return await fn();
      } catch (err) {
        if (attempt === this.opts.maxRetries || !isRetryable(err)) throw err;
        const delay = this.delayMs(attempt);
        await new Promise(r => setTimeout(r, delay));
      }
    }
    throw new Error("unreachable");
  }
}
Recommendation: Use decorrelated jitter for AI provider adapters. It is the strategy AWS employs in their own SDKs and provides the best convergence under high contention.

04 Circuit Breaker Pattern

Retries help with transient failures. But when a provider is genuinely down, retries waste resources and amplify load. The Circuit Breaker prevents calls to a failing service, giving it time to recover.

States

CLOSED

Requests flow through normally. Failure counter increments on each error. If failures exceed threshold, transition to OPEN.

OPEN

All requests fail fast immediately. No network call is made. After recovery timeout, transition to HALF-OPEN.

HALF-OPEN

A probe request is allowed through. Success → CLOSED. Failure → OPEN (resetting timeout).

Circuit Breaker State Machine
Request success Request failure Probe request Fast-fail (Open) State: CLOSED Failures: 0 / 5 Successes: 0

Full Implementation

TypeScript CircuitBreaker.ts
type CircuitState = "CLOSED" | "OPEN" | "HALF_OPEN";

interface CircuitBreakerOptions {
  failureThreshold: number;     // failures to trip OPEN
  recoveryTimeoutMs: number;   // time before HALF_OPEN probe
  successThreshold?: number;    // probes to close (default 1)
  isFailure?: (err: unknown) => boolean;
}

export class CircuitBreaker {
  private state: CircuitState = "CLOSED";
  private failures = 0;
  private successes = 0;
  private lastFailureTime: number | null = null;
  private onStateChange?: (s: CircuitState) => void;

  constructor(
    private opts: CircuitBreakerOptions,
    callbacks?: { onStateChange?: (s: CircuitState) => void }
  ) {
    this.onStateChange = callbacks?.onStateChange;
    this.opts.successThreshold ??= 1;
    this.opts.isFailure ??= (err) => true;
  }

  getState(): CircuitState { return this.state; }

  private transition(to: CircuitState): void {
    if (this.state === to) return;
    this.state = to;
    this.onStateChange?.(to);
  }

  async execute<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === "OPEN") {
      if (Date.now() - (this.lastFailureTime ?? 0) >= this.opts.recoveryTimeoutMs) {
        this.transition("HALF_OPEN");
        this.successes = 0;
      } else {
        throw new Error("Circuit breaker is OPEN");
      }
    }

    try {
      const result = await fn();
      this.onSuccess();
      return result;
    } catch (err) {
      this.onFailure();
      throw err;
    }
  }

  private onSuccess(): void {
    if (this.state === "HALF_OPEN") {
      this.successes++;
      if (this.successes >= (this.opts.successThreshold ?? 1)) {
        this.failures = 0;
        this.transition("CLOSED");
      }
    } else {
      this.failures = Math.max(0, this.failures - 1);
    }
  }

  private onFailure(): void {
    this.failures++;
    this.lastFailureTime = Date.now();
    if (this.state === "HALF_OPEN" || this.failures >= this.opts.failureThreshold) {
      this.transition("OPEN");
    }
  }
}

Notice how the breaker wraps the adapter call rather than modifying it. This is the decorator pattern in practice — the same contract, with enhanced behavior.

05 Failover & Fallback

When one provider fails, another can take its place. The primary/secondary pattern lets you route to a backup model (e.g., GPT-4 → Claude → local Llama) without changing upstream code.

Primary/Secondary Provider Pattern

TypeScript FailoverAdapter.ts
import type { ReplyEvent } from "@allternit/replies-contract";

interface ProviderAdapter {
  name: string;
  send(input: string): AsyncIterable<ReplyEvent>;
  healthCheck(): Promise<boolean>;
}

export class FailoverAdapter {
  constructor(
    private providers: ProviderAdapter[],
    private fallback?: (input: string) => AsyncIterable<ReplyEvent>
  ) {
    if (providers.length === 0) throw new Error("At least one provider required");
  }

  async *send(input: string): AsyncGenerator<ReplyEvent> {
    for (const provider of this.providers) {
      try {
        const healthy = await provider.healthCheck();
        if (!healthy) continue;

        yield* provider.send(input);
        return;
      } catch (err) {
        console.warn(`Provider ${provider.name} failed:`, err);
      }
    }

    if (this.fallback) {
      yield* this.fallback(input);
    } else {
      throw new Error("All providers exhausted and no fallback configured");
    }
  }
}

Graceful Degradation

Not every failure needs a model switch. Sometimes you can degrade the quality of the response:

  • Streaming → Batch: If the stream aborts, fall back to a non-streaming completion call.
  • Tools → No tools: If tool-use fails, retry with the same prompt but no tool definitions.
  • Rich → Plain: If citations/sources fail, return plain text without attribution.

Health Checks

A health check should be cheap and representative. For AI providers, a good pattern is a lightweight /models or /health endpoint call. Avoid using a full completion as a health probe — it burns tokens and rate limit budget.

TypeScript healthCheck.ts
async function providerHealthCheck(baseUrl: string, timeoutMs = 3000): Promise<boolean> {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), timeoutMs);
  try {
    const res = await fetch(`${baseUrl}/health`, { signal: controller.signal });
    return res.ok;
  } catch {
    return false;
  } finally {
    clearTimeout(timer);
  }
}

06 Observability

You cannot improve what you cannot measure. Resilience patterns add complexity; observability tells you whether they are working.

Adapter Lifecycle Hooks

Hook into the adapter at key points to emit structured telemetry:

TypeScript ResilientAdapter.ts
interface AdapterMetrics {
  requestCount: number;
  errorCount: number;
  retryCount: number;
  circuitState: CircuitState;
  latencyMs: number[];
  tokensConsumed: number;
}

export class ResilientAiSdkAdapter extends AiSdkReplyAdapter {
  private metrics: AdapterMetrics = {
    requestCount: 0, errorCount: 0, retryCount: 0,
    circuitState: "CLOSED", latencyMs: [], tokensConsumed: 0
  };

  private record(event: string, meta?: Record<string, unknown>) {
    console.log(`[adapter] ${event}`, {
      runId: this.runId,
      ts: Date.now(),
      ...meta
    });
  }

  async consume(stream: AsyncIterable<AiSdkStreamPart>): Promise<void> {
    const start = Date.now();
    this.metrics.requestCount++;
    this.record("consume.start");

    try {
      await super.consume(stream);
      this.metrics.latencyMs.push(Date.now() - start);
      this.record("consume.success", { latencyMs: Date.now() - start });
    } catch (err) {
      this.metrics.errorCount++;
      this.record("consume.error", { error: (err as Error).message });
      throw err;
    }
  }

  getMetrics(): AdapterMetrics {
    return { ...this.metrics };
  }
}

  • Throughput (requests/sec): Track using a rolling window counter. Sudden drops often precede error spikes.
  • Latency percentiles (p50, p95, p99): Average latency hides tail behavior. p99 tells you when rate limiting or retries are hurting real users.
  • Error rate by category: 429s suggest rate-limit misconfiguration; 503s suggest provider degradation; timeouts suggest network issues.
  • Circuit state transitions: Each OPEN transition is an alert. Frequent oscillation between OPEN and HALF-OPEN means your threshold or timeout is mis-tuned.
Tip: Export metrics in a format your orchestrator can scrape. The ServiceStatus contract from layer-boundary-contracts.ts includes a health field (healthy | unhealthy | degraded). Use adapter metrics to drive that status.
