MCP Adapter — Model Context Protocol

Uses @modelcontextprotocol/sdk to expose plugins as MCP tools. Each plugin function becomes a tool. Supports stdio, HTTP, and WebSocket transports.

TypeScript
class McpAdapter implements Adapter {
  name = 'mcp';
  version = '1.0.0';

  canActivate() {
    // MCP works anywhere Node.js runs
    return typeof process !== 'undefined';
  }

  async initialize(plugin, config) {
    return new McpAdapterInstance(plugin, config);
  }
}

CLI Adapter — Terminal Interface

Parses process.argv, executes the plugin once, prints results, and exits. Supports JSON, markdown, and text output modes. Designed for Codex, Aider, and shell workflows.

TypeScript
class CliAdapter implements Adapter {
  name = 'cli';

  async start() {
    await this.plugin.initialize(this.host);
    const args = this.parseArgs();
    await this.runOnce(args); // one-shot execution
  }
}

Native Adapter — Full Allternit Runtime

The richest implementation. Checks for globalThis.__ALLTERNIT_RUNTIME__ and wires every capability directly to the platform. Full LLM, tools, UI panels, persistent storage, and session history.

TypeScript
class NativeAdapter implements Adapter {
  name = 'allternit';

  canActivate() {
    return typeof globalThis.__ALLTERNIT_RUNTIME__
      !== 'undefined';
  }
}
export interface Adapter {
  readonly name: string;
  readonly version: string;

  // Can this adapter run here?
  canActivate(): boolean;

  // Create an instance for a plugin
  initialize(
    plugin: Plugin,
    config?: Record<string, any>
  ): Promise<AdapterInstance>;
}

Every adapter must answer three questions:

  • Who am I? — A name like "mcp" or "http" and a version.
  • Can I run here? — canActivate() inspects the environment. Native checks for __ALLTERNIT_RUNTIME__. MCP checks for process. CLI always returns true.
  • How do I start? — initialize() creates an AdapterInstance that wires the plugin to the platform.
The Contract

The PluginHost Interface

The PluginHost is the universal contract between a plugin and its runtime environment. It has six pillars: platform identity, LLM access, tool registry, UI capabilities, storage backend, and session context.

TypeScript src/types.ts — PluginHost
export interface PluginHost {
  readonly platform: string;     // "mcp" | "http" | "cli" ...
  readonly version: string;

  readonly llm: LlmCapabilities;
  readonly tools: ToolRegistry;
  readonly ui: UiCapabilities;
  readonly storage: StorageBackend;
  readonly session: SessionContext;
  readonly logger: Logger;
  readonly config: Record<string, any>;
}

How Adapters Map the Host

Each adapter creates its own implementation of PluginHost. The key insight: the same plugin runs unchanged because it only talks to the PluginHost interface, never to the platform directly.

// MCP host: LLM is delegated to Claude
class McpPluginHost implements PluginHost {
  platform = 'mcp';
  llm = new McpLlmCapabilities();
  tools = new McpToolRegistry();
  ui = new McpUiCapabilities();
  storage = new McpStorageBackend();
}

// HTTP host: LLM calls external APIs
class HttpPluginHost implements PluginHost {
  platform = 'http';
  llm = new HttpLlmCapabilities();  // calls OpenAI/Anthropic
  tools = new HttpToolRegistry();   // 7 tools registered
  storage = new HttpStorageBackend(); // in-memory + persistent flag
}

Think of PluginHost as a power strip with six outlets. Each adapter builds its own power strip, but the plugs (your plugin) are always the same shape.

  • MCP plugs into Claude Desktop — LLM requests are delegated to the host Claude instance. Storage is in-memory only (isPersistent = false).
  • HTTP plugs into the open web — LLM calls external APIs directly. Storage claims persistence but currently uses a Map (extensible to Redis/DB).
  • Native plugs into Allternit itself — every capability is a direct passthrough to the platform runtime.

Adapter Spotlight

MCP Adapter — Under the Hood

The MCP adapter is the most complex because it bridges two protocols: the Plugin SDK's synchronous-style interface and MCP's request/response server model.

TypeScript src/adapters/mcp/index.ts
class McpAdapterInstance implements AdapterInstance {
  private setupHandlers(): void {
    // 1. ListTools — exposes plugin functions as MCP tools
    this.server.setRequestHandler(ListToolsRequestSchema, async () => {
      const tools = this.plugin.manifest.provides.functions.map(fn => ({
        name: fn.name,
        description: fn.description,
        inputSchema: fn.parameters,
      }));
      return { tools };
    });

    // 2. CallTool — execute the plugin function
    this.server.setRequestHandler(CallToolRequestSchema, async (req) => {
      const result = await this.plugin.execute(
        req.params.name,
        req.params.arguments,
        { executionId: `mcp_${Date.now()}`, startedAt: new Date(), metadata: { source: 'mcp' } }
      );
      return { content: [{ type: 'text', text: result.content }] };
    });

    // 3. ListResources — manifest + readme
    this.server.setRequestHandler(ListResourcesRequestSchema, async () => {
      return {
        resources: [
          { uri: `plugin://${id}/manifest`, mimeType: 'application/json' },
          { uri: `plugin://${id}/readme`, mimeType: 'text/markdown' },
        ]
      };
    });
  }
}
Key insight: The MCP adapter registers three request handlers: ListTools (discovery), CallTool (execution), and ListResources (documentation). The plugin itself never knows it's talking to MCP — it just sees a standard PluginHost.
Adapter Spotlight

HTTP Adapter — Under the Hood

The HTTP adapter is the most flexible. It spins up an Express server with CORS, optional API key auth, rate limiting, SSE streaming, and WebSocket support.

TypeScript src/adapters/http/index.ts
private setupRoutes(): void {
  const router = Router();

  router.get('/health', (req, res) => {
    res.json({ status: 'ok', plugin: this.plugin.manifest.id });
  });

  router.post('/execute/:functionName', async (req, res) => {
    if (req.body.options?.stream) {
      return this.handleStreamingExecution(req, res, ...);
    }
    const result = await this.plugin.execute(
      req.params.functionName,
      req.body.inputs,
      { executionId: `http_${Date.now()}`, metadata: { source: 'http', ip: req.ip } }
    );
    res.json({ success: result.success, content: result.content, ... });
  });

  router.post('/batch', async (req, res) => {
    const results = await Promise.all(
      req.body.jobs.map(job => this.plugin.execute(job.function, job.inputs, ...))
    );
    res.json({ results });
  });

  this.app.use('/v1', router);
}
// SSE Streaming handler
private handleStreamingExecution(req, res, fnName, inputs) {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');

  const sendEvent = (event, data) => {
    res.write(`event: ${event}\n`);
    res.write(`data: ${JSON.stringify(data)}\n\n`);
  };

  sendEvent('start', { function: fnName });
  this.plugin.execute(...).then(result => {
    sendEvent('complete', result);
    res.end();
  });
}

When a client asks for streaming, the HTTP adapter switches protocols:

  • Normal request: Returns a single JSON response after execution finishes.
  • Streaming request: Opens an SSE (Server-Sent Events) connection. Sends a start event immediately, then a complete event when the plugin finishes. The client receives real-time progress without polling.

The WebSocket path does the same thing bidirectionally: clients can send execute messages and receive result messages over a persistent socket.

CLI Adapter — Under the Hood

The CLI adapter is the simplest: parse arguments, execute once, print, exit. But simplicity is its superpower — it works in any terminal, any CI pipeline, any Codex conversation.

TypeScript src/adapters/cli/index.ts
private parseArgs(): Record<string, any> {
  const args: Record<string, any> = {};
  const argv = process.argv.slice(2);

  for (let i = 0; i < argv.length; i++) {
    if (argv[i].startsWith('--')) {
      const key = argv[i].slice(2);
      const value = argv[i + 1];
      if (value && !value.startsWith('--')) {
        try { args[key] = JSON.parse(value); }
        catch { args[key] = value; }
        i++;
      } else {
        args[key] = true;
      }
    }
  }
  return args;
}
Design note: The CLI adapter parses --key value pairs and auto-detects JSON. --input '{"topic":"AI"}' becomes a real object, not a string. This lets Codex pass structured data without quoting hell.

Platform Native

Native Adapter — The Gold Standard

The Native adapter is what all others aspire to. It requires the Allternit runtime to be present, but in exchange it provides every capability with zero translation loss.

TypeScript src/adapters/native/index.ts
export interface AllternitRuntime {
  llm: {
    complete(prompt: string, options?: any): Promise<string>;
    stream(prompt: string, handler: (chunk: string) => void): Promise<void>;
    getModels(): string[];
  };
  tools: {
    execute(toolId: string, params: any): Promise<any>;
    list(): string[];
  };
  ui: {
    sendMessage(content: string, type?: string): void;
    createArtifact(type: string, content: any): string;
    openPanel(title: string, content: any): { update: Function; close: Function };
    showForm(schema: any): Promise<any>;
    progress(percent: number): void;
  };
  storage: { get(key: string): Promise<any>; set(key: string, value: any): Promise<void>; };
  session: { id: string; getHistory(): Promise<any[]>; };
}
class NativeAdapter implements Adapter {
  canActivate() {
    return typeof globalThis.__ALLTERNIT_RUNTIME__
      !== 'undefined';
  }
}

class NativeLlmCapabilities implements LlmCapabilities {
  async complete(prompt, options) {
    return this.runtimeLlm.complete(prompt, options);
  }
  async stream(prompt, handler, options) {
    return this.runtimeLlm.stream(prompt, handler);
  }
}

The Native adapter is a thin proxy. It doesn't reimplement anything — it just forwards calls:

  • LLM → directly to the Allternit runtime's LLM provider.
  • Tools → runtime.tools.execute() with no wrapper.
  • UI → runtime.ui.createArtifact() for images, code, charts, tables.
  • Storage → runtime.storage.get/set() with real persistence.
  • Session → runtime.session.getHistory() returns actual conversation history.

If the runtime exists, this is the adapter you want. If not, the others provide graceful fallbacks.

Group Chat: The Adapters Discuss Graceful Degradation

What happens when a plugin asks for a capability the platform doesn't support? Here's how the adapters handle it:

N
Native Adapter
I have everything: panels, persistent storage, full tool registry, session history. My plugins get the VIP treatment. 🏆
H
HTTP Adapter
I have SSE streaming and a REST API. But my UI is basically console.log — no real panels. I tell plugins isPersistent = true even though I'm just a Map for now. Fake it till you make it. 😅
M
MCP Adapter
Claude handles my LLM calls — I just delegate. But tools? I have to tell the plugin "ask the host, I don't have them." And storage is in-memory only. When the MCP server restarts, everything resets. 💨
C
CLI Adapter
I'm the bare minimum. No tools, no panels, no images, no persistent storage. But I work everywhere — CI, Docker, Codex, Aider. Sometimes simplicity wins. ⚡
N
Native Adapter
That's why the PluginHost contract has has(), isPersistent, and fallback modes. Plugins check before they leap. Smart plugins degrade gracefully — they simulate research if web_search isn't available.
Best practice: Always check host.tools.has('tool_id') before executing. If unavailable, fall back to LLM-based simulation or return a clear error. Never assume a capability exists.

Build Your Own Adapter

You now understand every built-in adapter. The ultimate test: implement a new one from scratch.

🎯 Project: WebSocket-First Real-Time Adapter

Create a new WebSocketAdapter that does the following:

  • Protocol: Pure WebSocket — no HTTP fallback. Clients connect to ws://localhost:3001.
  • Discovery: On connect, the server sends a manifest message with all available functions and their schemas.
  • Execution: Clients send { type: 'execute', function: 'name', inputs: {} }. The server streams progress updates via { type: 'progress', percent: 25 } messages before sending the final { type: 'result', ... }.
  • PluginHost: Implement a minimal host with in-memory storage, a console-based UI, and an LLM capability that delegates to an OpenAI API key passed in config.
  • Graceful degradation: If the plugin requests renderImage() but the client hasn't signaled image support, degrade to a markdown link.
  • Bonus: Add a subscribe message type so clients can receive plugin events (not just function results).

Deliverable: A single TypeScript file websocket-adapter.ts that implements the Adapter and AdapterInstance interfaces, plus a tiny test client HTML page.

Stretch goal: Compare your WebSocket adapter to the built-in HTTP adapter. What are the trade-offs? When would you choose one over the other? Document your reasoning in a DESIGN_DECISIONS.md file.
