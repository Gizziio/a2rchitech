At boot, UnifiedPluginRegistry initializes built-in plugins into a Map<string, UnifiedPlugin>, then loads user-installed plugins from localStorage. Every plugin carries a source discriminant ('built-in' | 'office' | 'feature' | 'marketplace' | 'local') that determines how it is loaded at runtime.

TypeScript src/lib/plugins/unified-registry.ts 🟢 Production 
class UnifiedPluginRegistry {
  private plugins: Map<string, UnifiedPlugin> = new Map();

  constructor() {
    this.initializeBuiltInPlugins();
    this.loadInstalledPlugins();
  }

  private initializeBuiltInPlugins(): void {
    const builtInPlugins: UnifiedPlugin[] = [
      {
        id: 'image',
        name: 'Image',
        source: 'built-in',
        status: 'enabled',
        category: 'create',
        capabilities: ['text-to-image', 'upscale'],
        runtime: { isLoaded: true },
        /* ... 10 built-in plugins + Office + Feature ... */
      }
    ];
    for (const plugin of builtInPlugins) {
      this.plugins.set(plugin.id, plugin);
    }
  }

  private loadInstalledPlugins(): void {
    if (typeof window === 'undefined') return;
    const installed = localStorage.getItem(STORAGE_KEYS.INSTALLED);
    if (installed) {
      const parsed: UnifiedPlugin[] = JSON.parse(installed);
      for (const plugin of parsed) {
        // Don't override built-in plugins
        if (!this.plugins.has(plugin.id)
            || plugin.source === 'marketplace'
            || plugin.source === 'local') {
          this.plugins.set(plugin.id, plugin);
        }
      }
    }
  }
}
Key insight: The registry uses a two-phase init: hard-coded built-ins first (guaranteed available), then user-installed plugins from localStorage. Built-ins can never be overridden by marketplace or local plugins — a security boundary.

Manifest parsing is handled by the SDK's utils.validateManifest(), but the platform re-validates at load time. Why? Because registry data can be stale, corrupted, or tampered with during transit. The platform treats the registry as a hint, not a source of truth.

TypeScript packages/@allternit/plugin-sdk/src/index.ts 🟢 Production 
export const utils = {
  validateManifest(manifest: any): manifest is PluginManifest {
    return (
      typeof manifest === 'object' &&
      typeof manifest.id === 'string' &&
      typeof manifest.name === 'string' &&
      typeof manifest.version === 'string' &&
      typeof manifest.runtime === 'string' &&
      typeof manifest.provides === 'object'
    );
  },

  async loadManifest(path: string): Promise<PluginManifest> {
    const fs = await import('fs/promises');
    const content = await fs.readFile(path, 'utf-8');
    return JSON.parse(content);
  }
};
Design principle: Trust but verify. The registry tells the platform where to look, but the platform validates the manifest itself before loading. This prevents registry poisoning attacks where a malicious entry points to a compromised plugin path.

Inside the platform, the runtime switch is even more granular. The UnifiedPluginRegistry uses a switch (plugin.source) to determine how to load each plugin's runtime instance:

TypeScript src/lib/plugins/unified-registry.ts 🟢 Production 
private async loadPluginRuntime(plugin: UnifiedPlugin): Promise<void> {
  if (plugin.runtime?.isLoaded) return;

  try {
    switch (plugin.source) {
      case 'built-in':
        const module = await import(`./${plugin.id}/plugin`);
        plugin.runtime = {
          instance: module.default || module.createPlugin?.(),
          isLoaded: true,
        };
        break;

      case 'office':
        const { OFFICE_PLUGINS } = await import('@/plugins/office-plugins');
        const officePlugin = OFFICE_PLUGINS.find(p => p.id === plugin.id);
        if (officePlugin) {
          plugin.runtime = { instance: officePlugin, isLoaded: true };
        }
        break;

      case 'feature':
        const { featureRegistry } = await import('@/plugins/feature.registry');
        const feature = featureRegistry.get(plugin.id);
        if (feature) {
          plugin.runtime = { instance: feature, isLoaded: true };
        }
        break;

      case 'marketplace':
        // Downloaded and loaded dynamically
        plugin.runtime = { isLoaded: true };
        break;

      case 'local':
        // Loaded from filesystem (dev mode)
        plugin.runtime = { isLoaded: true };
        break;
    }
  } catch (err) {
    throw new Error(`Failed to load plugin ${plugin.id}: `
      + (err instanceof Error ? err.message : 'Unknown error'));
  }
}
Watch the fallback: Notice that marketplace and local plugins currently have placeholder loaders (isLoaded: true with no instance). In production, these would integrate with a sandboxed module loader or Web Worker to prevent untrusted code from accessing the main thread.
Security in Practice

How the Platform Validates Permissions Before Execution

Before any plugin tool runs, the platform passes the request through policy gates. These are composable security checks: rate limiting, action allowlists, component restrictions, state validation, prop sanitization, and resource limits.

TypeScript adapters-ts/allternit-ix/src/runtime/policy-gates.ts 🟢 Production 
export interface PolicyGate {
  name: string;
  check: (
    type: 'action' | 'state-change' | 'render',
    context: PolicyContext
  ) => Promise<PolicyResult> | PolicyResult;
}

export function createActionAllowlistGate(
  allowedActions: string[]
): PolicyGate {
  const allowedSet = new Set(allowedActions);
  return {
    name: 'action-allowlist',
    check: (type, context) => {
      if (type !== 'action') return { allowed: true };
      if (!context.actionId || !allowedSet.has(context.actionId)) {
        return {
          allowed: false,
          reason: `Action '${context.actionId}' is not in allowlist`,
        };
      }
      return { allowed: true };
    },
  };
}

The platform also enforces prop sanitization to prevent XSS when plugins render UI components. String props are stripped of <script> tags, event handlers, and javascript: URLs before reaching the renderer.

TypeScript policy-gates.ts — Prop Sanitization 🟢 Production 
export function createPropSanitizationGate(options: {
  allowHTML?: boolean;
  allowedTags?: string[];
}): PolicyGate {
  const { allowHTML = false, allowedTags = [] } = options;
  return {
    name: 'prop-sanitization',
    check: (type, context) => {
      if (type !== 'render' || !context.props) return { allowed: true };

      const sanitizedProps = { ...context.props };
      let modified = false;

      for (const [key, value] of Object.entries(context.props)) {
        if (typeof value === 'string') {
          let sanitized = value;
          if (!allowHTML) {
            sanitized = sanitized.replace(/<[^>]*>/g, '');
          } else {
            // Remove script tags completely
            sanitized = sanitized.replace(/<script[^>]*>[\s\S]*?<\/script>/gi, '');
            // Remove event handlers
            sanitized = sanitized.replace(/on\w+\s*=/gi, '');
            // Remove javascript: URLs
            sanitized = sanitized.replace(/javascript:/gi, '');
          }
          if (sanitized !== value) {
            sanitizedProps[key] = sanitized;
            modified = true;
          }
        }
        // Block dangerous props
        if (key.startsWith('on') && typeof value === 'string') {
          delete sanitizedProps[key];
          modified = true;
        }
      }
      return { allowed: true, context: modified
        ? { ...context, props: sanitizedProps }
        : context };
    },
  };
}

Policy gates are composed into a single enforcement function. The default policy stack includes rate limiting (60 actions/minute), prop sanitization, and resource limits (1MB state, 100 components, 1000 actions).

TypeScript policy-gates.ts — Composition 🟢 Production 
export function composePolicyGates(...gates: PolicyGate[]): PolicyGate {
  return {
    name: `composed(${gates.map(g => g.name).join(', ')})`,
    async check(type, context) {
      let currentContext = context;
      for (const gate of gates) {
        const result = await gate.check(type, currentContext);
        if (!result.allowed) return result;
        if (result.context) currentContext = result.context;
      }
      return { allowed: true, context: currentContext };
    },
  };
}

export function createDefaultPolicyGates(): PolicyGate[] {
  return [
    createRateLimitGate({ maxActionsPerMinute: 60 }),
    createPropSanitizationGate({ allowHTML: false }),
    createResourceLimitGate({
      maxStateSize: 1024 * 1024, // 1MB
      maxComponents: 100,
      maxActions: 1000,
    }),
  ];
}
Error Handling at Scale

What Happens When a Plugin Crashes?

The platform isolates plugin failures at three levels: execution-level (try/catch around plugin.execute()), runtime-level (graceful unload on load errors), and registry-level (status rollback so a failed plugin doesn't stay "enabled").

TypeScript src/lib/plugins/plugin-runtime.ts 🟢 Production 
private async executeBuiltIn(
  pluginId: PluginId,
  input: string,
  files?: File[],
  context?: PluginExecutionRequest['context']
): Promise<PluginExecutionResult> {
  const startTime = Date.now();
  try {
    const plugin = await loadPlugin(pluginId);
    const result = await plugin.execute({ prompt: input, files, context });
    return {
      success: result.success,
      content: result.content || '',
      source: 'built-in',
      pluginId,
      actions: [{
        type: 'plugin-execute',
        description: `Execute ${pluginId} plugin`,
        timestamp: startTime,
      }],
      artifacts: result.artifacts?.map(a => ({
        type: a.type, name: a.name, url: a.url,
      })),
      deterministic: false,
    };
  } catch (err) {
    const error = err instanceof Error ? err : new Error(String(err));
    return {
      success: false,
      content: '',
      source: 'built-in',
      pluginId,
      actions: [],
      error: {
        code: 'PLUGIN_EXECUTION_ERROR',
        message: error.message,
        recoverable: false,
      },
    };
  }
}

Notice the structured error response: every failure returns a PluginExecutionResult with success: false, an error code, a human-readable message, and a recoverable flag. The chat UI uses recoverable to decide whether to show a "Retry" button.

TypeScript src/lib/plugins/unified-registry.ts — Status Rollback 🟢 Production 
private async setStatus(id: string, status: PluginStatus): Promise<boolean> {
  const plugin = this.plugins.get(id);
  if (!plugin) return false;

  const oldStatus = plugin.status;
  plugin.status = status;

  try {
    if (status === 'enabled') {
      await this.loadPluginRuntime(plugin);
    } else if (status === 'disabled') {
      await this.unloadPluginRuntime(plugin);
    }
    this.persistEnabledState();
    this.notifyListeners();
    return true;
  } catch (err) {
    // CRITICAL: Rollback on failure
    plugin.status = oldStatus;
    plugin.runtime = {
      isLoaded: false,
      loadError: err instanceof Error ? err.message : 'Unknown error',
    };
    return false;
  }
}

The deterministic executor adds another layer: it wraps the entire LLM-backed execution in a try/catch and returns a structured DeterministicResponse with an audit trail of every action taken before the failure.

TypeScript src/lib/plugins/deterministic-executor.ts 🟢 Production 
async execute(request: DeterministicRequest): Promise<DeterministicResponse> {
  this.startTime = Date.now();
  this.actions = [];

  try {
    // 1. Load command definition
    const loaded = await loadCommandMarkdown(request.pluginId, request.commandId);
    if (!loaded) {
      return this.createError(
        'COMMAND_NOT_FOUND',
        `Command ${request.commandId} not found in plugin ${request.pluginId}`,
        false
      );
    }

    // 2. Validate inputs
    const validation = this.validateInputs(loaded.command, request.inputs);
    if (!validation.valid) {
      return this.createError('INVALID_INPUTS', validation.error!, true);
    }

    // 3. Build system prompt and execute
    const systemPrompt = this.buildSystemPrompt(loaded);
    const result = await this.executeWithLLM(systemPrompt, request);

    return {
      success: true,
      output: result.output,
      actions: this.actions,
      artifacts: result.artifacts,
    };
  } catch (err) {
    const error = err instanceof Error ? err : new Error(String(err));
    return this.createError('EXECUTION_ERROR', error.message, false);
  }
}
Recovery pattern: Every error path preserves the actions array (audit trail). Even if the LLM call throws, the platform knows exactly which files were read, which APIs were called, and how far execution progressed before the crash. This is critical for debugging vendor plugins.

Step 1: Command Parsing

TypeScript src/lib/plugins/vendor-integration.ts 🟢 Production 
export function parseCommandInput(input: string): ParsedCommand {
  const trimmed = input.trim();
  // Vendor command format: /plugin:command
  if (trimmed.startsWith('/') && trimmed.includes(':')) {
    const [pluginPart, ...rest] = trimmed.slice(1).split(':');
    const commandPart = rest.join(':');
    const trigger = `/${pluginPart}:${commandPart}`;
    if (commandRegistry.has(trigger)) {
      return {
        type: 'vendor-command',
        pluginId: commandRegistry.get(trigger)!.pluginId,
        commandId: commandRegistry.get(trigger)!.name,
        raw: trimmed,
      };
    }
  }
  return { type: 'natural-language', raw: trimmed };
}

Step 2: PluginRuntime Routing

TypeScript src/lib/plugins/plugin-runtime.ts 🟢 Production 
async execute(request: PluginExecutionRequest): Promise<PluginExecutionResult> {
  const { input } = request;

  // 1. Check for explicit vendor command: /legal:triage-nda
  if (input.trim().startsWith('/') && input.includes(':')) {
    return this.executeVendorCommand(input, files, context);
  }

  // 2. Check for built-in mode trigger: /research, /code, etc.
  const parsed = parseCommandInput(input);
  if (parsed.type === 'built-in' && parsed.pluginId) {
    return this.executeBuiltIn(parsed.pluginId as PluginId, input, files, context);
  }

  // 4. Natural language — no specific plugin
  return {
    success: true,
    content: input,
    source: 'natural-language',
    actions: [],
  };
}

Quiz 1 / 3

Test Your Knowledge

What happens if a plugin's adapter type isn't supported by the platform?
Remember: Each adapter implements canActivate() as an environment probe. McpAdapter checks for process, NativeAdapter checks for globalThis.__ALLTERNIT_RUNTIME__. If no adapter can activate, the plugin has no valid host and initialization fails. There's no automatic fallback — the platform is explicit.
Quiz 2 / 3

Test Your Knowledge

Why does the platform re-validate manifests at load time rather than trusting the registry?
Defense in depth: The registry is a discovery service, not a trust anchor. The platform validates manifests locally using utils.validateManifest() before any code is loaded. This prevents registry poisoning attacks where a malicious entry points to a compromised plugin path.
Quiz 3 / 3

Test Your Knowledge

In the tool call flow, which component handles timeout cancellation?
Look at the type: The SDK's ExecutionContext includes an optional abortSignal?: AbortSignal. The platform's chat executor passes this signal from the user's cancellation action (clicking "Stop"). Adapters that support cancellation check abortSignal.aborted and throw an AbortError to cleanly terminate long-running LLM calls or tool executions.
