The Contract

The Plugin Manifest

Every Allternit plugin carries a manifest — a JSON contract that tells the host what the plugin is, what it needs, and what it provides. Without a valid manifest, a plugin cannot be registered, discovered, or executed.

Required Fields

The manifest is defined by the PluginManifest interface in types.ts. These fields are non-negotiable:

Field Type Purpose
id required string Unique kebab-case identifier (e.g. data-connector-v2)
name required string Human-readable display name
version required string Semantic version (major.minor.patch)
runtime required 'allternit-plugin-v1' Runtime spec version for compatibility
description required string Brief explanation for users and search
category required enum create | analyze | build | automate | research | custom
requires required HostRequirements LLM, tools, UI, and storage needs
provides required PluginFunctions Functions, parameters, and return schemas
adapters required AdapterConfig Entry points for each platform adapter
keywords optional string[] Search tags for registry discovery
config optional ConfigSchema User-configurable settings schema

Schema Validation

The SDK exports utils.validateManifest() — a runtime guard that checks structural correctness before a plugin is accepted into the registry:

TypeScript src/index.ts 
export const utils = {
  validateManifest(manifest: any): manifest is PluginManifest {
    return (
      typeof manifest === 'object' &&
      typeof manifest.id === 'string' &&
      typeof manifest.name === 'string' &&
      typeof manifest.version === 'string' &&
      typeof manifest.runtime === 'string' &&
      typeof manifest.provides === 'object'
    );
  },

  async loadManifest(path: string): Promise<PluginManifest> {
    const fs = await import('fs/promises');
    const content = await fs.readFile(path, 'utf-8');
    return JSON.parse(content);
  },

  async packagePlugin(sourceDir: string, outputPath: string): Promise<void> {
    // Would create .allp file
    console.error(`Packaging ${sourceDir} -> ${outputPath}`);
  },
};
Pro tip: Always validate manifests at build time (with a JSON Schema linter) and at runtime (with utils.validateManifest()). Build-time catches authoring errors; runtime catches tampered or corrupted packages.
SemVer

Versioning Strategy

Plugins are distributed as independent packages. Their versions communicate compatibility promises to hosts, users, and other plugins. Allternit mandates Semantic Versioning 2.0 for every plugin in the registry.

SemVer for Plugins

Schema version field format 
MAJOR.MINOR.PATCH

1.4.2
│   │   └── Bug fixes, security patches, docs updates
│   └── New functions, new adapters, new config options (backward-compatible)
└── Breaking changes: removed functions, changed parameters, new required permissions

Breaking vs Non-Breaking Changes

Change Type Version Bump Examples
Breaking MAJOR Remove a function, rename a parameter, add a required permission, change return schema shape
Feature MINOR Add a new function, add an optional adapter, add optional config field, expand keywords
Fix PATCH Fix runtime error, tighten validation, update docs, patch dependency vulnerability

Dependency Resolution

Plugins can declare dependencies on host capabilities (via requires) and on other plugins (via a future dependsOn field). The registry resolves these using a satisfies check:

TypeScript Dependency resolution logic 
function satisfies(
  requiredVersion: string,
  installedVersion: string
): boolean {
  // '^1.2.3' matches >=1.2.3 <2.0.0
  // '~1.2.3' matches >=1.2.3 <1.3.0
  const req = parseRange(requiredVersion);
  return req.test(installedVersion);
}

function resolvePluginDeps(
  manifest: PluginManifest,
  registry: PluginRegistry
): ResolutionResult {
  const missing: string[] = [];
  const incompatible: string[] = [];

  for (const dep of manifest.dependsOn || []) {
    const installed = registry.get(dep.id);
    if (!installed) {
      missing.push(dep.id);
    } else if (!satisfies(dep.version, installed.manifest.version)) {
      incompatible.push(`${dep.id}@${installed.manifest.version}`);
    }
  }

  return { ok: missing.length === 0 && incompatible.length === 0, missing, incompatible };
}
Warning: A plugin that bumps its requires field (e.g. adding a new mandatory tool like vector_search) is a breaking change for hosts that lack that tool. Always bump MAJOR when you increase host requirements.
Discovery

The Plugin Registry

The PluginRegistry class in the SDK is the local in-memory registry. But production systems need a remote registry — a central authority where hosts discover, search, and filter plugins by capability, trust level, and compatibility.

How Allternit Discovers Plugins

  1. Local scan: The host calls registry.loadFromDirectory('./plugins') to load built-in plugins.
  2. Remote fetch: The host queries GET /v1/registry/search?q=... for community plugins.
  3. Capability matching: The host filters results where plugin.requires is a subset of host.capabilities.
  4. Trust verification: Signed plugins are preferred; unsigned plugins trigger a warning.
  5. Install & activate: The host downloads the .allp package, validates it, and calls plugin.initialize(host).

Registry API Design

A production registry exposes these REST endpoints:

OpenAPI-style Registry API Specification 
// Search and filter plugins
GET /v1/registry/search?q={query}&category={cat}&adapter={adapter}&signed={true|false}
→ 200 OK
  { results: PluginSummary[], total: number, page: number }

// Get full manifest + metadata
GET /v1/registry/plugins/{pluginId}
→ 200 OK
  { manifest: PluginManifest, downloadUrl: string, checksum: string, signature: string }

// Download the .allp package
GET /v1/registry/plugins/{pluginId}/download?version={semver}
→ 200 OK (binary stream)

// Submit a new plugin (publisher auth required)
POST /v1/registry/plugins
  Body: { manifest: PluginManifest, package: base64, signature: string }
→ 201 Created | 409 Conflict (id+version exists)

// Get compatibility report for a host
POST /v1/registry/compat
  Body: { hostCapabilities: HostRequirements, pluginId: string }
→ 200 OK
  { compatible: boolean, missingTools: string[], missingUi: string[] }

Search & Filtering

Registry queries support multi-dimensional filtering:

  • Text search: Matches id, name, description, and keywords.
  • Category filter: Narrow to analyze, build, etc.
  • Adapter filter: Only show plugins that support mcp, http, native, etc.
  • Capability filter: Exclude plugins that require tools the host doesn't have.
  • Trust tier: verified (signed by Allternit), community (signed by author), unverified (no signature).
Architecture note: The registry is just another service. You can run a private registry behind your firewall, or mirror the public registry for offline deployments. The API spec remains the same.

Stage Breakdown

1. Build

The build step compiles TypeScript, bundles assets, and generates the .allp package. A valid build produces three artifacts:

  • dist/index.js — compiled plugin entry point
  • manifest.json — validated manifest (extracted from source)
  • package.allp — signed tarball of the above + README + LICENSE

2. Validate

Validation gates enforce structural and policy rules:

  • Schema validation: Manifest conforms to PluginManifest JSON Schema.
  • Runtime check: manifest.runtime === 'allternit-plugin-v1'.
  • Permission audit: Requested tools exist in the global tool registry.
  • Code scan: Static analysis flags eval(), new Function(), unauthorized network calls.
  • Size limit: Package must be < 50MB (configurable).

3. Sign

Code signing proves authorship and guarantees package integrity. The publisher signs the package digest with their private key; the registry verifies it against the author's public key.

TypeScript Signing flow 
import { createSign, createVerify } from 'crypto';

async function signPackage(
  packageBuffer: Buffer,
  privateKey: string
): Promise<string> {
  const signer = createSign('RSA-SHA256');
  signer.update(packageBuffer);
  return signer.sign(privateKey, 'base64');
}

function verifyPackage(
  packageBuffer: Buffer,
  signature: string,
  publicKey: string
): boolean {
  const verifier = createVerify('RSA-SHA256');
  verifier.update(packageBuffer);
  return verifier.verify(publicKey, signature, 'base64');
}

4. Publish

The signed package is uploaded to the registry. The registry:

  1. Checks that id + version is unique (or matches a deprecation policy).
  2. Re-verifies the signature against the publisher's registered public key.
  3. Stores the package in blob storage and updates the search index.
  4. Generates a CDN URL for global distribution.

5. Notify

Subscribed hosts receive a webhook or SSE event: plugin.updated or plugin.published. Hosts can then prompt users to install or auto-update based on their trust settings.

Security

Security Model

Plugins run arbitrary code inside host environments. Without a strong security model, a malicious plugin could exfiltrate data, abuse tools, or crash the host. Allternit's security architecture has four layers.

1. Permission System

The manifest's requires field is the declaration of intent. Every tool, UI renderer, and storage capability must be explicitly requested:

TypeScript Host capability check 
function checkPermissions(
  manifest: PluginManifest,
  host: PluginHost
): PermissionResult {
  const granted: string[] = [];
  const denied: string[] = [];

  for (const tool of manifest.requires.tools || []) {
    const toolId = typeof tool === 'string' ? tool : tool.tool;
    const required = typeof tool === 'string' ? true : tool.required;

    if (host.tools.has(toolId)) {
      granted.push(toolId);
    } else if (required) {
      denied.push(toolId);
    }
  }

  return {
    ok: denied.length === 0,
    granted,
    denied,
    // If denied has items, the plugin cannot initialize
  };
}

2. Sandboxing

Adapters act as sandbox boundaries. Each adapter decides how strictly to isolate the plugin:

  • Native: Runs inside the Allternit runtime process; isolation depends on the host OS (VMs, seccomp, or process boundaries).
  • HTTP: Plugin runs as a separate Node.js process; network access is controlled by firewall rules.
  • MCP: Plugin runs as a child process communicating over stdio; the MCP client (Claude Desktop, etc.) controls process lifetime.
  • CLI: Plugin shares the terminal process; minimal isolation — trust is derived from the package signature.
Never run unsigned plugins in the CLI adapter. Because CLI shares the terminal process, a malicious plugin has direct access to process.env, file system APIs, and shell execution. Always verify signatures before CLI execution.

3. Code Signing

Signing guarantees two properties: Authenticity (who wrote this?) and Integrity (was it tampered?). The registry stores each publisher's public key. When a host downloads a plugin, it:

  1. Downloads the package and its detached signature.
  2. Fetches the publisher's public key from the registry's /keys endpoint.
  3. Recomputes the package hash and verifies the signature.
  4. Rejects the plugin if verification fails — no exceptions.

4. Audit Trails

Every plugin action is logged for forensic analysis:

  • Install log: When, where, and by which user a plugin was installed.
  • Execution log: Which functions were called, with what parameters (sanitized), and how long they took.
  • Tool log: Every host.tools.execute() call, including success/failure and token usage.
  • Update log: Version transitions, rollback events, and signature verification results.
TypeScript Audit event structure 
interface AuditEvent {
  eventId: string;
  timestamp: Date;
  pluginId: string;
  pluginVersion: string;
  action: 'install' | 'activate' | 'execute' | 'update' | 'uninstall';
  actor: string;        // userId or system
  details: Record<string, any>;
  signatureValid: boolean;
}

State Transitions

Transition Trigger What Happens
Install User clicks "Add Plugin" Download .allp → verify signature → extract → validate manifest → write to ~/.allternit/plugins/
Activate Host starts or user enables Load JS → instantiate class → call plugin.initialize(host) → register functions
Deactivate User disables or host shuts down Stop accepting calls → call plugin.destroy() → unload from memory
Update New version published Deactivate old → download new → verify → activate new → migrate state if compatible
Uninstall User removes Deactivate → delete files → clear storage → remove from registry → log audit event

State Persistence

Plugins that rely on host.storage need guarantees about data survival across lifecycle transitions. The StorageBackend interface provides this contract:

TypeScript Storage contract 
interface StorageBackend {
  get<T = any>(key: string): Promise<T | undefined>;
  set<T = any>(key: string, value: T): Promise<void>;
  delete(key: string): Promise<void>;
  keys(): Promise<string[]>;
  clear(): Promise<void>;
  readonly isPersistent: boolean;
}

// Persistence guarantees by adapter
NativeAdapter   → isPersistent = true   // Disk-backed, survives restarts
HttpAdapter     → isPersistent = true   // In-memory Map (production: Redis/DB)
McpAdapter      → isPersistent = false  // Memory only, lost on restart
CliAdapter      → isPersistent = false  // Memory only, lost on exit
Migration rule: When updating a plugin, the host should copy storage keys that match ${oldPluginId}:* to ${newPluginId}:* if the MAJOR version hasn't changed. On MAJOR bumps, storage is isolated to prevent schema mismatches.

Graceful Degradation on Update

If an update fails mid-flight (network drop, signature invalid, incompatible requirements), the host must roll back to the previous version:

  1. Keep the old plugin instance alive until the new one passes initialize().
  2. If initialization fails, discard the new package and restore the old.
  3. Never leave the host in a state with zero working versions of a critical plugin.

Test Your Knowledge

When is a plugin update considered "breaking"?
Rule of thumb: If a host that worked with v1.x.y will stop working or error out after updating to v2.0.0, it's a breaking change. Always bump MAJOR when removing functions, renaming parameters, or adding new required host capabilities.
Quiz 3 / 3

Test Your Knowledge

Why code-sign plugins instead of just hashing?
Deep dive: A SHA-256 hash tells you "this file hasn't changed." A signature tells you "this file hasn't changed and it was created by the owner of private key X." In a plugin ecosystem, knowing who shipped the code is as important as knowing what was shipped.

🎯 Deliverable 5: Validation Function

Write a TypeScript function that validates a DataConnector manifest:

TypeScript validateDataConnectorManifest.ts 
import { PluginManifest, utils } from '@allternit/plugin-sdk';

type DataSource = 'postgres' | 'mysql' | 'rest' | 's3' | 'csv';

interface DataConnectorManifest extends PluginManifest {
  dataSources: DataSource[];
}

export function validateDataConnectorManifest(
  manifest: any
): { valid: boolean; errors: string[] } {
  const errors: string[] = [];

  // Base SDK validation
  if (!utils.validateManifest(manifest)) {
    errors.push('Invalid base PluginManifest structure');
    return { valid: false, errors };
  }

  // Category must be 'automate' or 'custom'
  if (manifest.category !== 'automate' && manifest.category !== 'custom') {
    errors.push(`Category must be 'automate' or 'custom', got '${manifest.category}'`);
  }

  // Must declare at least one data source
  const validSources: DataSource[] = ['postgres', 'mysql', 'rest', 's3', 'csv'];
  if (!Array.isArray(manifest.dataSources) || manifest.dataSources.length === 0) {
    errors.push('dataSources must be a non-empty array');
  } else {
    for (const ds of manifest.dataSources) {
      if (!validSources.includes(ds)) {
        errors.push(`Invalid data source: '${ds}'`);
      }
    }
  }

  // Must provide at least one function
  const fns = manifest.provides?.functions;
  if (!Array.isArray(fns) || fns.length === 0) {
    errors.push('Must provide at least one function');
  }

  // Connection config schema check
  const cfg = manifest.config?.properties;
  if (!cfg?.connectionString || cfg.connectionString.type !== 'string') {
    errors.push("config must include 'connectionString' of type string");
  }
  if (cfg?.connectionString && !cfg.connectionString.secret) {
    errors.push("connectionString must be marked as secret");
  }

  return { valid: errors.length === 0, errors };
}
