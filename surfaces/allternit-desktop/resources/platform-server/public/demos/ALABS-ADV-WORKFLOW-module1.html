01 The Problem

Most automation tools are either too rigid (linear sequence, no branching) or too chaotic (event soup with no execution guarantees). When you need to:

  • Fetch data from three APIs in parallel, then join the results
  • Branch execution based on a condition at runtime
  • Retry a failed node without restarting the whole flow
  • Visualize what actually happened during execution

...you need a directed acyclic graph (DAG) execution engine with first-class scheduling, state tracking, and visualization. Not a script. Not a chain of webhooks. An engine.

That's what @allternit/workflow-engine is — and by the end of this module, you will understand exactly how it executes workflows, schedules tasks, and renders DAGs.

02 The Concept: DAG Architecture

A workflow in Allternit is a DAG: a collection of nodes (steps) connected by edges (data flow), with no cycles. This guarantees that execution always terminates and that every node executes only after its dependencies are satisfied.

  ],

    { id: 'c2', source: 'fetch',   target: 'filter' },
    { id: 'c3', source: 'filter',  target: 'output' },
  ],
}

The engine uses this definition to build a DAG, identify source nodes (nodes with no incoming edges), and begin execution from there. As each node completes, the engine traverses outgoing connections and evaluates any conditional guards before triggering downstream work.

04 The Engine: DAG Execution

When you call engine.execute('data-pipeline', inputs), here is exactly what happens inside workflow-engine.ts:

🔧 TypeScript
async function execute(workflowId, inputs) {
  const workflow = workflows.get(workflowId);
  const dag = buildDAG(workflow);
  const execution = { id: uuidv4(), status: 'running', ... };

  const startNodes = dag.nodes.filter(n =>
    !dag.connections.some(c => c.target === n.id)
  );

  await Promise.all(
    startNodes.map(n => executeNode(n, execution, dag))
  );
}
🗣 Plain English

1. Look up the workflow definition by its ID. If it doesn't exist, throw immediately.

2. Build a DAG from the nodes and connections array. This is a shallow struct — just references.

3. Create an Execution record with a fresh UUID, copy inputs into context.variables, and initialize empty tracking arrays for active, completed, and failed nodes.

4. Find every node that has zero incoming connections. These are the starting points.

5. Fire all start nodes concurrently with Promise.all. The engine doesn't sequence them unless data dependencies force it.

Node Execution & Dependency Resolution

Each node's executeNode call is recursive. It checks dependencies, runs the executor, stores results, then walks outgoing edges:

🔧 TypeScript
async function executeNode(node, execution, dag) {
  const incoming = dag.connections
    .filter(c => c.target === node.id);

  const depsMet = incoming.every(c =>
    execution.context.state.completedNodes
      .includes(c.source)
  );
  if (!depsMet) return;

  execution.context.state.activeNodes
    .push(node.id);

  const result = await nodeType.executor(node, context, inputs);

  execution.context.state.nodeResults[node.id] = result;
  execution.context.state.completedNodes
    .push(node.id);

  for (const conn of outgoing) {
    if (conn.condition && !evaluateCondition(...))
      continue;
    await executeNode(nextNode, execution, dag);
  }
}
🗣 Plain English

1. Dependency Check: Before running, the node verifies that every node connected to its input has already completed. If not, it returns silently — it will be called again when the upstream node finishes.

2. Activation: The node ID is pushed into activeNodes so observers (and the visualizer) know it's running.

3. Input Assembly: Inputs are gathered from upstream nodeResults via connection mappings, then merged with any explicit InputMapping expressions.

4. Execution: The NodeType.executor function is invoked. This is where the actual work happens — HTTP requests, transforms, delays.

5. Result Storage: The return value is stored in nodeResults under the node's ID, and the node moves from activeNodes to completedNodes.

6. Downstream Trigger: For each outgoing connection, the engine evaluates the optional condition string. If true (or absent), it recursively calls executeNode on the target.

Key Insight: Execution State is a Journal

The ExecutionState interface tracks everything as arrays and maps — not a database, not an event log, just a plain object mutated in-place:

  • activeNodes — what's running right now
  • completedNodes — what finished successfully
  • failedNodes — what threw an error
  • nodeResults — the output data from each node
  • executionPath — the order nodes were activated

This design makes the engine observable by default. Hooks like beforeNodeExecute and afterNodeExecute receive the live context and can stream it to any visualization layer.

05 The Scheduler: Parallel Execution

The engine handles logic. The scheduler handles concurrency. It is a task queue with configurable limits, timeouts, and priority sorting.

🔧 TypeScript
function createScheduler(config = {}) {
  const {
    maxConcurrency = 5,
    queueLimit = 100,
    defaultTimeout = 60000,
    enablePriority = false,
  } = config;

  const queue = [];
  const running = new Map<string, AbortController>();

  async function processQueue() {
    if (isPaused || running.size >= maxConcurrency
        || queue.length === 0) return;

    if (enablePriority)
      queue.sort((a, b) => b.priority - a.priority);

    const task = queue.shift();
    const controller = new AbortController();
    running.set(task.id, controller);

    try {
      const timeoutId = setTimeout(
        () => controller.abort(),
        task.timeout || defaultTimeout
      );
      const result = await executeTask(task, controller.signal);
      clearTimeout(timeoutId);
      completed++;
    } finally {
      running.delete(task.id);
      processQueue();
    }
  }
}
🗣 Plain English

1. Configuration: You set a concurrency ceiling (default 5), a queue cap (100), and a per-task timeout (60s). Priority sorting is opt-in.

2. Queue: Tasks are plain objects with an ID, a node reference, and an execution context. When you submit(), the task enters the queue.

3. Processing Loop: processQueue() is called after every submit and after every completion. It is a recursive pull, not a polling loop.

4. Concurrency Guard: If running.size >= maxConcurrency, the function returns immediately. The task waits in queue.

5. AbortController: Every running task gets an AbortController. The scheduler can cancel tasks via controller.abort(), and the executor can check signal.aborted to clean up.

6. Cascade: The finally block always calls processQueue() again, ensuring the next queued task starts as soon as a slot opens.

06 The Visualizer: Layout & Export

The visualizer turns a workflow definition into geometry. It uses topological sorting with layering to compute node positions automatically, then generates SVG paths, Mermaid syntax, or Graphviz DOT.

🔧 TypeScript
function autoLayoutNodes(nodes, connections) {
  const incoming = new Map();
  const outgoing = new Map();

  connections.forEach(conn => {
    outgoing.get(conn.source)?.push(conn.target);
    incoming.get(conn.target)?.push(conn.source);
  });

  // Kahn's algorithm: topological sort by layer
  const layers = [];
  while (visited.size < nodes.length) {
    const layer = nodes.filter(n =>
      !visited.has(n.id) &&
      (inDegree.get(n.id) || 0) === 0
    );
    layers.push(layer);
    layer.forEach(n => {
      visited.add(n.id);
      outgoing.get(n.id)?.forEach(t => {
        inDegree.set(t, inDegree.get(t) - 1);
      });
    });
  }

  // Position each layer horizontally centered
  layers.forEach((layer, i) => {
    const layerWidth = layer.length * (w + gap) - gap;
    const startX = (canvasWidth - layerWidth) / 2;
    layer.forEach((node, j) => {
      positions[node.id] = {
        x: startX + j * (w + gap),
        y: 100 + i * (h + gap)
      };
    });
  });
}
🗣 Plain English

1. Build Adjacency Lists: For every connection, record which node points to which. This gives us in-degree (how many prerequisites) and out-degree (how many dependents) for each node.

2. Kahn's Algorithm: In each iteration, collect every node whose in-degree is zero. These nodes have no unresolved dependencies, so they can all be placed in the same horizontal layer.

3. Decrement In-Degrees: After placing a layer, "remove" those nodes from the graph by decrementing the in-degree of every node they point to. This exposes the next layer.

4. Position: Each layer is centered on the canvas. Nodes within a layer are spaced evenly. Vertical distance between layers is constant. The result is a clean, left-to-right or top-to-bottom flow with no edge crossings when possible.

5. Paths: Connections are rendered as cubic Bézier curves from the bottom of the source node to the top of the target node, with control points at the vertical midpoint.

Export Targets

The visualizer is not just for display. It is a format converter:

  • toSVG() — Generates a self-contained SVG with arrow markers, colored rectangles, and labels. Embed directly in documentation.
  • toMermaid() — Outputs graph TD syntax. Paste into GitHub markdown, Notion, or any Mermaid renderer.
  • toDOT() — Outputs Graphviz DOT. Use with dot -Tpng for high-resolution static renders.

All three share the same layout() call, so the geometry is identical across formats.

07 Execution Lifecycle

A workflow execution moves through a finite state machine. Understanding these states is critical for building reliable automation:

pending running completed failed paused cancelled timeout
Execution State Machine
type ExecutionStatus =
  | 'pending'    // Workflow created, not yet started
  | 'running'    // Nodes are actively executing
  | 'paused'     // Execution suspended (retains state)
  | 'completed'  // All reachable nodes finished without error
  | 'failed'     // A node threw and errorHandling did not recover
  | 'cancelled'  // Cancelled by user or parent workflow
  | 'timeout'    // Exceeded defaultTimeout (5 min default);

// State transitions are one-way except pause ↔ running
// failed, cancelled, and timeout are terminal

Hooks: Observing Execution

The engine accepts a hooks object in its config. These are async functions called at precise lifecycle moments:

  • beforeExecute — Called once after the execution record is created but before any node runs
  • beforeNodeExecute — Called before every single node invocation
  • afterNodeExecute — Called after a node completes, receiving the result
  • afterExecute — Called once after the workflow reaches a terminal state
  • onError — Called when any node throws an uncaught error

Hooks are how you build logging, metrics, audit trails, and real-time UIs on top of the engine without modifying its core.

Custom Node Types

The built-in types (triggers, transforms, HTTP, conditions) are just pre-registered NodeType objects. You can register your own:

Custom Node Registration
engine.registerNodeType({
  type: 'custom:processor',
  category: 'custom',
  displayName: 'Data Processor',
  inputs:  [{ name: 'data', type: 'object', required: true }],
  outputs: [{ name: 'result', type: 'object' }],
  configSchema: [{ name: 'multiplier', type: 'number', default: 1 }],
  executor: async (node, context, inputs) => {
    const multiplier = node.config?.multiplier || 1;
    return { result: { value: inputs.data.value * multiplier } };
  },
});
