Core Idea (Normalized)

The Glide MD asserts:

Motion, transitions, and interactive affordances are not cosmetic —
they are explanatory instruments that reduce cognitive load.

In A2rchitech terms:
    •    Motion encodes state change
    •    Transitions encode causality
    •    Interactive UI encodes operability
    •    Visual continuity encodes trust

This maps perfectly onto:
    •    Presentation Kernel → InteractionSpec
    •    Canvas Protocol → renderer obligations
    •    Embodied Interaction Layer (already defined)

TO-BE Mapping (What this MD adds)

This MD extends, not replaces, existing specs.

1. Renderer Capability Declaration (New Concept)

We introduce a Renderer Capability Profile:

RendererCapabilities:
- supports_motion_semantics
- supports_transition_continuity
- supports_physics
- supports_interactive_explainers

This lets:
    •    CLI/TUI renderers degrade gracefully
    •    Web/Mobile renderers go full Glide-level polish
    •    Kernel adapt output without branching logic

➡️ Where this goes
    •    /spec/presentation/RendererCapabilities.md (new, small spec)
    •    Referenced by Presentation Kernel

⸻

2. Transition Semantics (Formalized)

We add a rule:

Transitions are semantic edges, not animations.

This becomes a constraint, not a feature.

Examples:
    •    capsule_spawned → tab slides in
    •    approval_granted → canvas unlocks
    •    run_completed → timeline resolves

➡️ Where this goes
    •    Patch to /spec/presentation/PresentationKernel.md
    •    Small addition to InteractionSpec

⸻

3. “Explainability via Motion” Rule

New invariant (compatible with SOT):

Any automated change SHOULD be explainable via visible transition,
unless renderer explicitly cannot support it.

➡️ Enforced at renderer level, not kernel level.
